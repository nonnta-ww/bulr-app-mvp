# bulr プロジェクト 引き継ぎドキュメント

このドキュメントは、bulr プロジェクトの戦略・コンセプト議論を新しいセッション（問診設計セッション、プロトタイプ実装セッション）に引き継ぐためのものです。新しいセッションを開始する際は、このドキュメントを最初に貼り付けてコンテキストを共有してください。

**最新の方向性**：bulr は「面接官の隣に座る AI 面接アシスタント」として位置付けられる（v2 から確定）。プロダクト体験の詳細は `bulr-product-direction.md` を参照。

## 2. 解くべき課題

### 中核課題

**AI時代における中堅以上エンジニアの「実務判断力」を可視化する手段が市場にない**

### 課題の構造

1. **フォーマット問題**：職務経歴書がフリーフォーマットで、構造化された比較ができない
2. **情報損失問題**：書く側の言語化能力と実務能力にズレがあり、書類選考で実力者が落ちる（特に書くのが苦手な優秀層）
3. **生産性問題**：採用担当のエンジニアが本業に時間を使えない（年間800書類レビューの工数）
4. **評価精度問題**：面接で実力を見抜けず、雰囲気で採用してミスマッチが起きる
5. **結果コスト問題**：エージェントフィー（20-35%）+ 早期離職による経営インパクト

### 既存サービスの限界

- **Findy / LAPRAS / Forkwell**：アウトプット（GitHub・技術ブログ・登壇）がある層しか可視化できない。だが情報発信するエンジニアは全体の数パーセント。戦力となるミドルレンジ層は構造的に取れない。
- **paiza / AtCoder**：コーディング能力の可視化に偏る。AI時代にコーディング自体の価値は相対的に低下。
- **SPI系**：地頭は測れるが、実務判断力は測れない。

### 創業者自身が体現する事例

創業者自身が「アウトプットしないが年収2000万近い実力者」であり、書類だけでは実力が伝わらない層の代表例。Microsoftで OS を書いていた友人も、書類を書くのが苦手で書類選考で落ちることが多い、という事例も把握。

## 3. プロダクトコンセプト

### コアコンセプト（v2 確定版）

**bulr は、面接官の隣に座る、AI 面接アシスタント**

エンジニア採用面接で、面接官の発言を最小限に抑えながら、候補者の実務判断力を引き出す質問を提案し、回答を構造化して評価する。書類選考に時間を奪われる採用責任者と、書類では実力が伝わらないエンジニアの両方を救う。

詳細なプロダクト設計は `bulr-product-direction.md` を参照。

### 何を測るか

「**AIを道具として、複合領域にまたがるシステムを成立させる総合力**」

#### 評価軸：広さ

- システム種別の経験（BtoC/BtoB SaaS/業務系/決済/組み込み/データ基盤など）
- トラフィック・データ規模の経験
- チーム・組織の経験（一人〜数百人）
- フェーズの経験（0→1, 1→10, 10→100, レガシー再生, 撤退）
- 職能横断の経験（企画/デザイン/ビジネス判断などへの関与）

#### 評価軸：深さ

- 設計判断の経験（アーキ選定、DB、API、責務分離）
- トラブル対応の経験（障害、性能、セキュリティ、データ破損）
- 負債との対峙（リファクタ、リプレイス、レガシー保守）
- 組織判断（採用、評価、技術文化、プロセス）
- AI活用の判断（どこをAIに任せ、どこを自分で判断するか）

#### 評価軸：意思決定の射程

タスク → 機能 → プロダクト → 事業 → 組織。射程が広い人ほど、AI時代に希少価値が高い。

### どう測るか

**対話型経験問診** を、面接官と候補者の対面コミュニケーションの中で実施する。bulr は録音から文字起こし → LLM 分析 → 次の質問候補3つを面接官に提案する。

- 第1段：「Xという状況に直面したことがあるか」（経験有無）
- 第2段：「具体的にどんな症状で、どう特定したか」（経験の真贋）
- 第3段：「どう解決したか、なぜそれを選んだか」（判断力）
- 第4段：「別の選択肢はあったか、規模が違ったら同じ判断をするか」（メタ認知）

経験者は淀みなく答えられる。経験浅い人は2-3段で詰まる。盛っている人は深掘りで矛盾が出る。書くのが苦手でも、対話なら経験を引き出せる。

### アウトプット形式

**多次元のスキルヒートマップ** + **サマリーテキスト**（面接後に面接官向けに生成、職種で分けず経験の濃淡を可視化）

例：「API設計とDB設計に経験が深い、要件定義もそこそこできる、フロントエンドは触ったことがある程度、AWSは設計まで一通り、組織マネジメントは10人規模まで」

### ドラクエのパーティ比喩（重要なメンタルモデル）

創業者の採用観：

1. 全部できる人は存在しないのに、求人票はそれを求める
2. スキル分布の解像度が低く、戦士に魔法を求めるミスマッチが起きる
3. クエスト（事業フェーズ）に応じてパーティを組み替えるべきなのに、組織は固定的

bulr は将来的に「ドラクエ的な組織編成インフラ」を目指す。ただし MVP では候補者個人の可視化に集中する。

## 4. ターゲット

### 主たる利用者（v2 確定）

**採用面接の面接官**：事業会社の VPoE / エンジニアリングマネージャー / 採用責任者、スタートアップの CTO や創業者、採用担当のシニアエンジニア。候補者は bulr を直接使わない。bulr は面接官の側に存在し、面接官の判断を支援する。

### 評価対象

- 中堅以上のエンジニア（実務経験5年以上目安）

### 顧客（マネタイズ対象、Stage 2 以降）

- 即戦力採用・中核ポジション採用に困っている事業会社
- AI時代を前提とした先進的な経営層
- 業務委託マッチング市場（将来）

### 構造的に外す

- ジュニア層・新卒（AI時代の構造変化により別市場）
- 既にアウトプットで可視化されている上位数パーセントの有名エンジニア（既存サービスで足りる）

## 5. 既存構想からの主要な変更点

### 元の構想（v0）

- 個人向け bulr + 企業向け bulr business の両面プロダクト
- 3種類の診断（地頭・仕事力・専門職）
- 選択式の知識テスト型診断
- バックエンド100項目・フロントエンド100項目で職種別に分割
- スキル診断 → 人材紹介 → 教育 のマネタイズ展開

### v1（候補者直接対話型）

- MVP は企業起点の B2B から（両面プラットフォームは2-3年後）
- 対話型経験問診に切り替え（選択式知識テストではない）
- 候補者が AI と直接 30〜40 分テキスト対話
- 職種で分けず、ヒートマップで濃淡可視化

### v2（面接アシスタント型、最終形）

- 主体を「面接官」に変更（候補者は bulr 画面を見ない）
- 入力モダリティを音声録音 + Whisper 文字起こしに変更
- LLM の役割を「対話相手」から「次の質問候補3つを面接官に提案」に変更
- 状態A（録音中）/ 状態B（候補選択）の 2 状態 UI
- 人材紹介事業を Stage 3 以降のオプションとして余地を残す
- マネタイズは2年目以降と割り切る
- 教育ビジネスには進まない

### v2 採用理由

- 対話品質のリスクを切り離せる（人間の面接官が対話の主役）
- 検証データの質が高い（候補者は人間相手で本気で答える）
- 市場が受け入れやすい（「人間が面接することが重要」観念に整合）
- 採用責任者の本当の痛み（「面接で何を聞けば実力が分かるか」）に直接刺さる
- 創業者の優位性（26年の経験、年間800書類）が最大化される

## 8. ビジネスモデルの方向性

### Year 1（最初の12ヶ月）：MVP と検証

- 面接アシスタント型のプロトタイプ（バックエンドエンジニア領域から）
- ベトナム人20-30 + 日本人10-20の面接でベータ協力面接官5-10人が実際に使う
- パートナー企業5社獲得（無料ベータ）
- ゴール：「bulr 問診を受けた候補者の採用後パフォーマンスが良い」というエビデンス作り

### Year 2：職種拡大とマネタイズ開始

- フロントエンド、SRE/インフラ、PdM寄り領域に拡張
- リアルタイム文字起こし + 先読みでの質問生成
- 話者分離 API（Deepgram など）の導入
- 受験者2000-5000人規模
- 企業向け月額（10-30万）でマネタイズ開始
- 業務委託マッチング領域への展開検討

### Year 3：両面化とインフラ化

- 候補者直接対話型の追加
- 企業内現有メンバーの可視化
- ドラクエ的パーティ編成シミュレーション
- 「bulr スコア」が業界用語化を狙う
- 他サービスが bulr API を組み込む段階へ
- **人材紹介事業の判断ポイント**（後述）

### Year 4-5：標準化と社会変革

- bulr スコアが業界標準に
- アトラクト支援、リテンション支援、組織編成コンサルティングなど派生事業
- 「採用」を超えた「人材流動性インフラ」へ

### マネタイズの方向性（仮説）

- **取らない**：教育コンテンツ作成（労力対効果悪い）
- **取る**：企業向け SaaS 月額、API 利用料、組織診断コンサル、データ販売
- **保留・条件付きで取る**：人材紹介（Stage 3 以降のオプション、後述）

## 9. 事業オプション（条件付きで余地を残すもの）

### 9.1 人材紹介事業

**位置づけ**：Stage 3 以降のオプション。今すぐやらない。やる時のために、データモデルとポジショニングは余地を残しておく。

**着想の背景**：
For Startups（証券コード7089）の2026年3月期上期決算で、売上高前年同期比33%増、営業利益110.3%増と大幅増収増益を達成しており、人材紹介市場の収益インパクトの大きさを再認識。

**For Startups の構造から学べること**：

- 彼らの強みは「人材紹介ビジネス単体」ではなく、「**スタートアップ特化 × データベース「STARTUP DB」（25,000社超） × タレントエージェンシー**」の3点セット。
- 参入障壁は「データベース」と「業界人脈」であり、人材紹介はマネタイズ手段の一つ。
- これは bulr が目指す方向と近い。bulr における STARTUP DB に相当するのが「対話型問診による実務判断力データベース」。

### 取るべき入り方（やる時の条件）

bulr のデータベースが充実してから、「**データを持っているからこそできる紹介**」として人材紹介を後付けで導入する。

具体的には Stage 3 以降で、「bulr 問診を受けた候補者の中から、貴社の求める判断力プロファイルに合う人を紹介します」という形。これは普通のエージェントには絶対できない（他社は実務判断力データを持っていないため）。

### 取ってはいけない入り方（避けるべき判断）

Stage 1〜2 の段階で人材紹介を始めること。理由は3つ：

1. データベースが薄い段階での人材紹介は、普通のエージェントと同じ
2. 人材紹介を始めた瞬間、bulr は中立的なインフラではなくなる
3. 人材紹介はオペレーション集約型のビジネスで、組織運営が必要

### Stage 1 で意識しておくこと

- データモデル設計時に、人材紹介を後付けできる構造（候補者を独立テーブルに分離する等）を考慮しておく
- ただし Stage 1 で実装はしない
- 競合状況（For Startups、Findy 転職、新興プレイヤー）を四半期ごとに確認

### 9.2 業務委託マッチング

**位置づけ**：Stage 2 以降の有力オプション。

業務委託は意思決定が速い、見極めニーズが強い、bulr の判断力可視化と相性が良い。Year 2 で展開を検討する。

### 9.3 アトラクト・リテンション支援

**位置づけ**：Stage 4 以降。

データベースが大規模化して初めて意味のあるベンチマーク提供が可能になる。早期に手を出さない。

## 10. 技術スタック（Stage 1 確定版）

### MVP プロトタイプ用の確定構成

- **モノレポ**：pnpm workspaces + Turborepo
- **フロント+API**：Next.js 16 (App Router) + React 19 + TypeScript
- **スタイリング**：Tailwind CSS 4 + shadcn/ui
- **AI**：Vercel AI SDK 6 (`generateObject` 中心) + Anthropic Claude API (Sonnet 4.6)
- **音声文字起こし**：OpenAI Whisper API
- **音声ストレージ**：Vercel Blob（30日後自動削除、Vercel Cron）
- **DB**：Neon Postgres + Drizzle ORM
- **認証**：Better Auth（マジックリンクのみ、面接官向け）+ Resend
- **デプロイ**：Vercel Hobby（個人アカウント、1プロジェクト）
- **管理画面**：apps/web 内の `/admin` ルート + ADMIN_ALLOWED_EMAILS 許可メール検査

### Stage 1 で導入しないもの

- apps/admin の分離（Stage 2 で）
- apps/business（Stage 3 で）
- リアルタイム文字起こし（Stage 2）、話者分離 API（Stage 2）、先読み質問生成（Stage 2）
- PostHog、Sentry、Helicone、Cloudflare R2、BetterStack
- pgvector
- i18n（日本語のみで運用）
- packages/auth、packages/ui の切り出し（apps/web 内に直書き）
- 候補者向けインターフェース（Stage 3 で追加検討）
- `useChat` / `streamText`（v1 はチャット UI 前提だったが v2 では使わない）

### ドメイン構成（長期）

- 個人向け：https://bulr.net
- 企業向け：https://bz.bulr.net
- API：https://api.bulr.net
- 管理：https://admin.bulr.net

Stage 1 では仮ドメインまたは bulr.net のみ。

## 11. 3ヶ月プロトタイプ計画

### 3ヶ月のゴール

**「バックエンドエンジニア向けの面接アシスタントを作り、創業者および協力面接官5-10人が、ベトナム人20-30人 + 日本人10-20人の面接で実際に使い、問診パターンに基づく面接結果と、面接官の独自判断との一致度を確認する」**

### 3ヶ月でやらないこと（明確に捨てる）

- ヒートマップの本格的な可視化（Stage 1 は簡易版で十分）
- 企業向けダッシュボード
- パーティ編成シミュレーション
- 組織編成機能
- 課金システム
- 複数職種対応
- マネタイズの設計
- ブランディング・ドメイン取得・会社設立
- 候補者向け UI

### Phase 1：問診設計の骨格作り（完了）

バックエンドエンジニアの実務判断力を構成する状況パターンを57個書き出す（`docs/02-questionnaire-patterns.md`）。

- 設計判断 D-01〜15 (15個)
- トラブル対応 T-01〜12 (12個)
- 性能・規模 P-01〜08 (8個)
- セキュリティ・データ S-01〜08 (8個)
- 組織・プロセス O-01〜08 (8個)
- AI 活用 A-01〜06 (6個)

### Phase 2：深掘り質問ロジックの設計（完了）

各パターンに対し、4段階の深掘り質問を設計（`docs/03-probe-logic.md`）。

### Phase 3：プロトタイプ実装の設計（進行中）

技術スタック確定済み（上述）、最低限機能の設計。Kiro spec-driven development で 6 spec を生成。

### Phase 4：実装開始 + パートナー打診

プロトタイプ実装着手、5-10人のパートナー候補に打診。

### 第2ヶ月

プロトタイプ完成、ベトナム人で面接を回す、回答を分析して問診を磨く。

### 第3ヶ月

日本人で面接を回す、結果と既知の実力評価を比較、相関を確認、3ヶ月後の意思決定。

### 3ヶ月後の意思決定ポイント

- 問診は機能しているか
- 企業は価値を感じているか
- 自分はこれをやり続けたいか

3つ揃えば本格投資、1つでも欠ければ方向転換。

## 12. リスクと留保

### 主要リスク

1. **音声録音 + Whisper の精度と運用コスト**（候補者の発話品質、雑音、複数人重複発話）
2. **質問設計をどうスケールさせるか**（創業者一人では限界、人脈活用が前提）
3. **質的情報を企業が読めるレポート形式に落とし込めるか**（ヒートマップ + サマリーテキストの設計が命）
4. **ベータ協力面接官の確保**（5-10人の確保）
5. **「先に行きすぎる」リスク**（過去の Twitter/Facebook より前の類似サービス失敗経験）

### 留保

- ジュニア層問題は社会課題として認識しつつ、bulr のスコープ外と割り切る
- マネジメント層の評価は別軸ではなく、システム成立経験の構成要素として扱う
- 人材紹介は Stage 3 以降のオプションとして温存
