## 1. プロダクトの一行説明

**bulr は、面接官の隣に座る、AI 面接アシスタント**

エンジニア採用面接で、面接官の発言を最小限に抑えながら、候補者の実務判断力を引き出す質問を提案し、回答を構造化して評価する。書類選考に時間を奪われる採用責任者と、書類では実力が伝わらないエンジニアの両方を救う。

## 2. プロダクトの基本コンセプト

### 2.1 何をするプロダクトか

bulr は、エンジニア採用の**面接プロセスに介入する AI ツール**である。具体的には次の3つを行う。

ひとつ、面接の前に、候補者の経歴に応じて「**今日聞くべき実務判断力テーマ**」を選定する。

ふたつ、面接の最中に、候補者の回答を分析し、「**次に聞くべき深掘り質問**」を3択で面接官に提案する。

みっつ、面接の後に、候補者の回答を分析し、「**実務判断力のヒートマップ**」と「**サマリーレポート**」を生成する。

### 2.2 誰が使うか

主たるユーザーは**採用面接の面接官**である。具体的には：

- 事業会社の VPoE / エンジニアリングマネージャー / 採用責任者
- スタートアップの CTO や創業者
- 採用担当のシニアエンジニア

候補者は bulr を直接使わない。bulr は面接官の側に存在し、面接官の判断を支援する。これは将来的に変わる可能性があるが、Stage 1 ではこの形を取る。

### 2.3 何を測るか

「**AIを道具として、複合領域にまたがるシステムを成立させる総合力**」

具体的には3つの軸で測る。

**広さの軸**

- システム種別の経験（BtoC/BtoB SaaS/業務系/決済/組み込み/データ基盤など）
- トラフィック・データ規模の経験
- チーム・組織の経験（一人〜数百人）
- フェーズの経験（0→1, 1→10, 10→100, レガシー再生, 撤退）
- 職能横断の経験（企画/デザイン/ビジネス判断などへの関与）

**深さの軸**

- 設計判断の経験（アーキ選定、DB、API、責務分離）
- トラブル対応の経験（障害、性能、セキュリティ、データ破損）
- 負債との対峙（リファクタ、リプレイス、レガシー保守）
- 組織判断（採用、評価、技術文化、プロセス）
- AI活用の判断

**意思決定の射程の軸**

- タスク → 機能 → プロダクト → 事業 → 組織

### 2.4 どう測るか

**対話型経験問診**を、面接官と候補者の対面コミュニケーションの中で実施する。

4段階の深掘り構造：

- 第1段：「Xという状況に直面したことがあるか」（経験有無）
- 第2段：「具体的にどんな症状で、どう特定したか」（経験の真贋）
- 第3段：「どう解決したか、なぜそれを選んだか」（判断力）
- 第4段：「別の選択肢はあったか、規模が違ったら同じ判断をするか」（メタ認知）

経験者は淀みなく答えられる。経験浅い人は2-3段で詰まる。盛っている人は深掘りで矛盾が出る。書くのが苦手でも、対話なら経験を引き出せる。

## 3. なぜ面接アシスタント型なのか

### 3.1 検討した2つの選択肢

bulr の実装形態として、2つのアプローチを検討した。

**選択肢A：候補者直接対話型**
候補者がサービスにログインし、LLM と直接対話して問診を受ける。

**選択肢B：面接アシスタント型**
面接官が bulr を起動し、候補者との対面面接の中で bulr の支援を受ける。

### 3.2 選択肢B（面接アシスタント型）を採用する理由

**理由1：対話品質のリスクを切り離せる**
候補者が直接 LLM と30-40分対話するモデルは、対話の自然さや UX 品質が完走率を左右する。LLM プロンプトの磨き込みに膨大な時間を取られる。

面接アシスタント型なら、人間の面接官が対話の主役なので、LLM が多少不器用な質問を出しても、面接官が咀嚼して投げ直せる。Stage 1 で検証したい本丸（「実務判断力を見抜ける問診パターンが設計できているか」）に集中できる。

**理由2：検証データの質が高い**
候補者が LLM と対話する場合、テキスト入力は疲れるし、対話相手が AI だと真剣度が下がる。面接官との対話なら、候補者は本気で答える。問診パターンの妥当性検証の素材として、面接アシスタント型の方が圧倒的にクリーン。

**理由3：市場が受け入れやすい**
2026年時点で「人間が面接することが重要」という企業観念は根強い。LLM 単独で実力評価する形は、企業側が「本当に信頼できるのか」と疑念を持つ。

面接アシスタントなら「人間の判断を補強する道具」として位置付けられ、導入心理的ハードルが低い。

**理由4：採用責任者の本当の痛みに直接刺さる**
創業者が年間800書類で痛感していた問題のうち、「面接で何を聞けば実力が分かるか分からない」「面接が雑談で終わって判断できない」という痛みは、面接アシスタント型が直接的に解決する。

**理由5：創業者の優位性が最大化される**
26年の経験、年間800書類、VPoE経験、これらが結晶化した「問診パターン」は、面接官の道具として使われる時に最も価値を発揮する。LLM が代行するのではなく、優れた面接官の知見を、普通の面接官にも使えるようにする、というポジショニング。

### 3.3 候補者直接対話型は捨てるのか

捨てない。Stage 2 以降のオプションとして残す。データベースが充実し、市場の受容度が上がってから追加する。長期的には両方持つ可能性がある。

## 4. プロダクト体験（面接中の UX）

### 4.1 設計原則

**面接官が画面を見なくても面接が成立する**

bulr は面接の主役ではなく、後ろで支援する黒子。画面を見るのは候補者の発話の隙間、または候補者が考え込んでいる数秒だけ。

### 4.2 面接フロー全体

```
面接前
  ├─ 面接官が bulr で新規セッション作成
  ├─ 候補者情報入力（名前、応募職種、経歴の要点）
  └─ bulr が問診パターンから優先度の高いものを提示

面接中（ループ）
  ├─ 状態A：質問表示 + 録音中
  │   └─ [次の質問へ] ボタン待ち
  │
  └─ 状態B：分析結果 + 次の質問候補
      └─ [1] [2] [3] [自分で次を聞く] のいずれかを選択
      └─ 状態A に戻る

面接後
  ├─ ヒートマップ生成
  └─ サマリーレポート生成
```

### 4.3 面接中の画面状態

**状態A：録音中**

```
┌─────────────────────────────────────┐
│  [現在の質問]                        │
│  N+1 問題に対処した経験はありますか?  │
│  具体的にどんな症状で、どう特定し     │
│  ましたか?                           │
│                                      │
│  [このセクションの目的]              │
│  DB 性能に対する判断力の確認         │
│                                      │
│  ● 録音中  経過: 1:23                │
│                                      │
│  進捗: 7/12 パターン、23分/40分      │
│                                      │
│           [ 次の質問へ ]             │
└─────────────────────────────────────┘
```

操作は [次の質問へ] のみ。

**状態B：分析結果 + 次の質問候補**

```
┌─────────────────────────────────────┐
│  [今のターンの文字起こし]（折り畳み）  │
│  候補者：ECサイトの商品一覧で N+1 が │
│  発生し、Datadog APM で特定...        │
│                                      │
│  [評価サマリー]                      │
│  ✓ N+1 経験あり、特定方法も具体的    │
│  △ 解決方法はまだ聞けていない        │
│  △ 規模・トレードオフは未確認        │
│                                      │
│  [次の質問候補]                      │
│  ① 解決策と選んだ理由を聞く          │
│  ② 規模が10倍なら同じ判断をしたか聞く │
│  ③ 次のパターン：API設計に進む       │
│                                      │
│  [ ① ] [ ② ] [ ③ ] [自分で次を聞く] │
└─────────────────────────────────────┘
```

操作は4つの選択肢のみ。

### 4.4 「自分で次を聞く」の挙動

3候補のいずれも選ばない場合、面接官は [自分で次を聞く] を押す。

このボタンを押した瞬間に**録音が即開始**される。面接官は質問を話す。候補者が回答する。[次の質問へ] を押す。

bulr は録音から「面接官の発話（質問）」と「候補者の発話（回答）」を LLM で抽出し、通常のターンと同じ形でデータを記録する。

これにより、面接官は **3候補を採用するか、自分の判断で進めるか**を自由に選べる。LLM の提案はあくまでオプションであり、強制されない。

### 4.5 状態遷移の特徴

画面状態が2つしかない。これが Stage 1 の UI 設計の重要な特徴。

- 状態A：録音中（操作1種類）
- 状態B：選択中（操作4種類）

面接官の認知負荷を最小化する。

## 5. 技術的な実現方式

### 5.1 録音と文字起こし

**ボタン押下時にまとめて Whisper API へ送信**

「次の質問へ」または「自分で次を聞く」を押した瞬間に録音を停止し、それまでの音声を OpenAI Whisper API に送信。文字起こし結果を LLM に渡して、評価と次の質問候補を生成する。

Stage 1 ではこの方式で開始する。回答が長くて待ち時間が出る問題は許容する（30秒程度の沈黙は面接の質を上げる側面もある）。

Stage 2 以降で、チャンク単位ストリーミング文字起こしや先読み生成の導入を検討する。

### 5.2 話者分離

**プロンプトで吸収する。専用 API は使わない。**

LLM は文脈から「質問」と「回答」を区別できる。短く疑問形の発話は質問、長く具体的な発話は回答、というパターンを LLM は容易に識別する。

プロンプトで明示的に「面接官の質問と候補者の回答を区別し、候補者の回答のみを評価対象にしてください」と指示する。

混在の懸念（面接官が話しすぎる、相槌の混入、被り発話）は Stage 1 のサンプル数（70人程度）では創業者の目視確認で十分カバーできる。Stage 2 で Deepgram などの専用 API 導入を検討する。

### 5.3 質問候補の生成

「次の質問へ」押下時、LLM が以下を判断して3候補を提示する。

候補1：**深掘りを続ける**
現在のパターンの4段階のうち、まだカバーできていない段階の質問を生成。

候補2：**メタ認知や別視点を問う**
現在のパターンの最終段階（規模感、代替案、振り返り）の質問を生成。

候補3：**次のパターンに進む**
現在のパターンが十分カバーされたと判断した場合、次に聞くべきパターンの第1段の質問を生成。

3候補のうち1つは必ず「次のパターンに進む」候補を含める。面接時間の管理を支援する。

## 6. Stage 1 の実装範囲

### 6.1 Stage 1 のゴール

**「バックエンドエンジニア向けの面接アシスタントを作り、創業者および協力面接官5-10人が、ベトナム人20-30人 + 日本人10-20人の面接で実際に使い、問診パターンに基づく面接結果と、面接官の独自判断との一致度を確認する」**

### 6.2 Stage 1 で作るもの

**問診パターン（最重要）**

- バックエンドエンジニア向けの状況パターン40-60件
- 各パターンに4段階の深掘り質問テンプレート
- 各パターンを「広さ × 深さ」の軸にマッピング

**面接セッション管理**

- 面接官のログイン（マジックリンク）
- セッション作成、候補者情報入力
- セッションの一覧、詳細表示

**面接中の UI**

- 状態A（録音中 + 質問表示）
- 状態B（分析結果 + 3候補提示 + 自分で次を聞く）

**音声処理**

- ブラウザでの録音
- Whisper API への送信
- 文字起こし結果の保存

**LLM 統合**

- 文字起こし結果の評価
- 次の質問候補3つの生成
- セッション完了時のヒートマップ生成

**面接後のレポート**

- ヒートマップ可視化（簡易版）
- サマリーテキスト
- 文字起こし全文の閲覧

**管理画面**

- 全セッションの一覧（創業者用）
- 各セッションの詳細、回答評価
- 手動評価の入力欄

### 6.3 Stage 1 で作らないもの

- 候補者向けのインターフェース全般
- ヒートマップの本格的な可視化（Stage 1 は簡易版で十分）
- 企業向けダッシュボード
- 求人管理、応募管理
- ワークスペース機能（マルチテナント）
- パーティ編成シミュレーション
- 課金システム
- 複数職種対応（バックエンドのみ）
- 音声の話者分離 API
- リアルタイム文字起こし
- 先読みでの質問生成
- 多言語対応（日本語のみ）
- モバイルアプリ

### 6.4 Stage 1 の検証指標

検証の成功条件：

ひとつ、**面接官が完走できるか**
予定時間内（30-40分）で12-15パターンをカバーできる。

ふたつ、**bulr の提案が役に立つか**
3候補の中から1以上を選ぶ率が70%以上。

みっつ、**評価の一致度が高いか**
bulr のヒートマップ評価と、面接官の独自判断の相関が高い。

よっつ、**面接官が継続したいと思うか**
ベータ参加面接官の70%以上が「使い続けたい」と回答。

これら4指標が揃えば、Stage 2 への本格投資を判断する。

## 8. プロダクト哲学

### 8.1 AI は黒子、人間が決める

bulr は AI が人間を置き換えるツールではない。優れた人間の判断を支援するツール。

UI の至るところで、この哲学を表現する。

- 「自分で次を聞く」が常に選択肢として存在する
- LLM の提案は3候補に絞り、面接官に選ばせる
- 最終評価には創業者の手動評価を組み合わせる

### 8.2 書類では伝わらない実力を救う

bulr の存在意義は、書類選考やアウトプットでは可視化されない実力者を可視化すること。情報発信が苦手なエンジニアが、面接で実力を発揮できる仕組みを作る。

これは創業者自身が体現する課題であり、Microsoft で OS を書いていた友人のような実例がある。

### 8.3 ドラクエのパーティ思考

エンジニア採用は「全部できる人を探す」のではなく「ドラクエのパーティを組む」ように考えるべき。bulr は将来的に、組織のパーティ編成を支援するインフラを目指す。

Stage 1 ではこのビジョンは UI には現れないが、データ設計はこの方向性を意識する。

### 8.4 AI時代の実務判断力に焦点を絞る

コードを書く能力は AI が代替する時代。bulr が測るのは「何を作るか、どう判断するか」という、AI時代に価値が上がる軸。

ジュニア層は構造的にターゲット外。中堅以上の判断力に集中する。

## 9. プロトタイプの作成方針

### 9.1 リポジトリ

- 個人アカウント / Private
- リポジトリ名：`bulr-app-mvp`
- 本格稼働時はリポジトリごと作り直す方針

### 9.2 技術スタック

- pnpm workspaces + Turborepo
- Next.js 16 + React 19 + TypeScript
- Tailwind CSS 4 + shadcn/ui
- Vercel AI SDK 6 + Anthropic Claude API（Sonnet）
- Drizzle ORM + Neon Postgres
- Better Auth（マジックリンク）+ Resend
- OpenAI Whisper API（音声文字起こし）
- Vercel Hobby（個人アカウント、1プロジェクト）

### 9.3 デプロイ構成

Stage 1 は Vercel に 1プロジェクトのみ。apps/web の中に面接官向け UI と管理画面（/admin ルート + Basic 認証）を統合する。

### 9.4 開発の進め方

3つのセッションに分けて進める。

**戦略・コンセプトセッション**
方向性の議論、事業判断、ピボット判断。このセッションが該当。

**問診設計セッション**
40-60件の問診パターンの設計、4段階深掘り質問の作成、問診の磨き込み。継続的に長く続く作業。

**プロトタイプ実装セッション**
リポジトリ初期化、技術実装、UI実装、デプロイ。コードベース中心の対話。

## 10. 未解決の論点

ここまでの議論で固まっていない、Stage 1 に入る前に決めておくべき論点。

### 10.1 候補者の同意と倫理

面接の録音と AI による評価は、候補者の同意が必要。Stage 1 でも明示的な同意フローを設けるか、ベータ協力者には事前に説明するか、どう運用するか。

### 10.2 ベータ協力面接官の確保

5-10人の協力面接官をどう集めるか。創業者の人脈から候補リストを作る作業が必要。

### 10.3 ベトナム人受験者の現実性

ベトナム500人サンプルが取得可能とのことだが、実際の面接を行う面接官は誰か、面接の言語は日本語/英語/ベトナム語のどれか、文化的なギャップで判断力評価がフェアにできるか、検討が必要。

### 10.4 問診パターンの第一版

着手済み（`docs/02-questionnaire-patterns.md`）。新セッション②（問診設計）で整理と深掘り質問の設計を進める。

### 10.5 ワイヤーフレーム

ここまでの UI 議論を、実際の画面として可視化する。手書きまたは Figma で、状態A・状態B の画面を作る。コードを書き始める前に必須の作業。

## 11. このドキュメントの使い方

このドキュメントは、bulr プロダクトとしての「現在の合意点」を記録する。

新しいセッション（問診設計セッション、プロトタイプ実装セッション）を開始する際は、このドキュメントを参照することで、戦略・実装の意思決定の整合性を保つ。

プロダクトの方向性に変更が生じた場合は、このドキュメントを更新する（v2、v3 と版数管理）。

戦略レベルの判断は引き継ぎドキュメント v2 を、技術アーキテクチャは Stage 1 / Stage 2 のアーキテクチャ資料を、それぞれ参照する。
