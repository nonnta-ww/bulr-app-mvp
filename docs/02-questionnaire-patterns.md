# bulr 問診設計：状況パターン集（Week 1 成果物）

このドキュメントは、bulr の対話型経験問診における「状況パターン」の初稿である。
バックエンドエンジニア領域における **実務判断の射程と深さを引き出すための状況** を 6 カテゴリ × 57 個に分けて記述する。

参考：`docs/bulr-handoff.md`（プロジェクト戦略全体）, `docs/backend-skills.csv`（旧路線：選択式知識テストの素材）, `docs/backend-question.md`（旧路線：質問試案）

---

## 0. 設計哲学

### 0.1 状況パターンとは何か

選択式の知識テスト（「○○を使ったことがあるか」「○○を選択してください」）ではなく、
**「ある状況に直面し、トレードオフを抱えて判断した経験」** を 1 単位とする。

| 旧路線（知識テスト型） | 新路線（状況パターン型） |
|---|---|
| EXPLAIN を使ったことがあるか？ | 本番でクエリ性能が突然劣化し、原因特定からインデックス設計の見直しまで判断した経験 |
| マイクロサービスの設計経験は？ | モノリスのうち 1 機能だけが他の数倍のリソースを要するようになり、分割するか同居で吸収するか判断した経験 |
| 認証機能を設計したことがあるか？ | 既存の認証方式が要件に合わなくなり、ユーザー影響を抑えながら置き換えた経験 |

### 0.2 良い状況パターンの条件

1. **状況が具体的**：「経験あり/なし」が明瞭に答えられる粒度
2. **判断点を含む**：単一の正解ではなく、トレードオフを伴う選択がある
3. **判別性がある**：本当に経験した者と聞きかじりの者で、深掘りで差が出る
4. **5 年以上の中堅以上が遭遇する蓋然性**：ジュニアでは出会いにくい状況

### 0.3 横断軸（深掘りで効く軸）

各パターンに対し、Week 2 の深掘り設計で以下を聞き分ける：

- **判断の射程**：タスク / 機能 / プロダクト / 事業 / 組織
- **規模文脈**：一人 / 数人 / 数十人 / 数百人、QPS、データ量、ユーザー数
- **システム種別**：BtoC / BtoB SaaS / 業務系 / 決済 / 組み込み / データ基盤
- **フェーズ**：0→1 / 1→10 / 10→100 / レガシー再生 / 撤退
- **AI 活用観点**：どこを AI に任せ、どこを自分で判断したか（AI 時代の希少価値の核）

### 0.4 パターン採番ルール

- `D-XX`：設計判断（Design）
- `T-XX`：トラブル対応（Trouble）
- `P-XX`：性能・規模（Performance）
- `S-XX`：セキュリティ・データ（Security）
- `O-XX`：組織・プロセス（Organization）
- `A-XX`：AI 活用・AI ネイティブ開発（AI）

## 1. 設計判断パターン（D, 15個）

中核は「**複数の妥当な選択肢があり、なぜそれを選んだか**」が問えるか。
構造的トレードオフを認識し、文脈に応じて意思決定した経験を引き出す。

### D-01: モノリス内の機能の独立サービス化判断
- **状況**: モノリスで動いている機能のうち、1 つが他の数倍のトラフィック・データ量・スケール要件を抱えるようになった。独立サービスとして切り出すか、モノリス内で吸収するかを判断した。
- **判断点**: 分割コスト（運用、観測性、デプロイ、データ整合）vs スケール限界 / 組織境界 / 障害分離
- **見極め**: 「分割すべきだった」「分割しすぎた」の両方の経験談を持っているか。境界の引き方の根拠を語れるか。
- **射程**: プロダクト〜組織

### D-02: 既存スキーマで窮屈になった新要件の取り込み
- **状況**: 既存の RDB スキーマに収めようとすると無理が出る新要件が来た。テーブル分割、意図的な非正規化、別ストア（NoSQL/検索/時系列）導入のいずれかを選択した。
- **判断点**: スキーマ整合性 vs 性能 / 開発速度 / 運用複雑性
- **見極め**: 「正規化の原則」と「現実のトレードオフ」をどう折り合わせたか。後悔した判断と、その学びを語れるか。
- **射程**: 機能〜プロダクト

### D-03: 同期処理の非同期化・疎結合化
- **状況**: 外部 API や重い処理に同期で依存している箇所が、サービス全体の可用性・性能の足かせになった。キュー導入、イベント駆動化、キャッシュなどで疎結合化を判断した。
- **判断点**: 同期の単純さ vs 非同期の信頼性・性能。整合性ウィンドウとユーザー体験のトレードオフ。
- **見極め**: 「結果整合で良い領域」と「同期で守るべき領域」の線引き根拠。失敗時の補償設計を持っているか。
- **射程**: 機能〜プロダクト

### D-04: API スタイル選択と後方互換性戦略
- **状況**: 新規 API、または既存 API の刷新で、REST / gRPC / GraphQL / イベントベース のいずれを選ぶか判断した。バージョニング・後方互換性の戦略も含む。
- **判断点**: クライアント特性 / チームスキル / 性能要件 / エコシステムのトレードオフ
- **見極め**: 「自分が好きな方式」ではなく文脈で選んでいるか。互換性破壊時の移行戦略を持っているか。
- **射程**: 機能〜プロダクト

### D-05: 認証・認可方式の選定／置き換え
- **状況**: 既存の認証方式（独自セッション、自前 OAuth など）が要件に合わなくなり、IdP 導入・SSO・トークン管理方式の置き換えを判断した。
- **判断点**: 自前運用 vs マネージド / セキュリティ vs UX / 移行コスト vs 将来負債
- **見極め**: ユーザー影響・既存トークンの扱い・移行期間の併存設計など、運用面の判断を含むか。
- **射程**: プロダクト〜事業

### D-06: トランザクション境界と整合性モデル
- **状況**: 複数のリソースに跨る処理で、ACID トランザクションでまとめるか、結果整合（Saga, Outbox など）で組むかを判断した。
- **判断点**: ビジネス上の整合性要件 vs スケーラビリティ / 障害復旧の単純さ
- **見極め**: 「ここはズレてはいけない」「ここは数秒ズレても大丈夫」の見極めを言語化できるか。補償処理の設計経験があるか。
- **射程**: 機能〜プロダクト

### D-07: バッチ vs ストリーミング / push vs pull
- **状況**: データ処理を、バッチで定期実行するか、ストリーミングで継続処理するか、また通知を push で出すか pull で取らせるか判断した。
- **判断点**: レイテンシ要件 / 複雑性 / 運用コスト / コスト構造
- **見極め**: 「ストリーミングの方が新しいから」ではなく、業務要件とのフィット根拠を持つか。
- **射程**: 機能〜プロダクト

### D-08: マルチテナント設計
- **状況**: SaaS 等で複数顧客のデータを扱う際、DB 分離 / スキーマ分離 / 行レベル分離のいずれを選ぶか判断した。
- **判断点**: 分離強度 vs 運用効率 / コスト / 移行容易性
- **見極め**: 顧客規模・規制要件・契約形態が判断にどう影響したか。
- **射程**: プロダクト〜事業

### D-09: 共通ライブラリ／プラットフォーム化の判断
- **状況**: 複数チーム・サービスで似た実装が現れ、共通ライブラリ化・社内プラットフォーム化を判断した。
- **判断点**: DRY のメリット vs 過剰抽象化 / 結合度上昇 / 維持コスト
- **見極め**: 「3 回現れたら共通化」のような自分の判断基準を持つか。失敗した抽象化の経験を語れるか。
- **射程**: プロダクト〜組織

### D-10: フロント / バック / BFF の責務分離
- **状況**: 機能を実装するにあたり、ロジックをフロント / バック / BFF のどこに置くかを判断した。
- **判断点**: クライアント多様性 vs サーバー集約 / セキュリティ境界 / レイテンシ
- **見極め**: 「ビジネスロジックをフロントに置かない」程度の通念を超え、文脈で判断できるか。
- **射程**: 機能〜プロダクト

### D-11: ドメインモデルと DB スキーマの距離設計
- **状況**: ドメインモデルと DB スキーマを直結させるか分離するか、ORM をどこまで信頼するかを判断した。
- **判断点**: 開発速度 vs 純粋性 / テスト容易性 / 性能透明性
- **見極め**: アクティブレコード派 / リポジトリ派 / DDD 派、それぞれの利点と限界を語れるか。
- **射程**: 機能〜プロダクト

### D-12: 信頼性設計（リトライ・冪等性・タイムアウト・DLQ）
- **状況**: 外部依存・非同期処理が絡むフローで、リトライ戦略・冪等性キー・タイムアウト・デッドレターキューを設計した。
- **判断点**: 「失敗を許容する設計」 vs 「失敗を防ぐ設計」 / 重複と欠損のどちらを許すか
- **見極め**: 「at-least-once / at-most-once / exactly-once」の意味を業務要件に紐づけて選択できるか。
- **射程**: 機能〜プロダクト

### D-13: 新規システムの DB スキーマをゼロから設計
- **状況**: 新規プロダクト・新機能で、DB スキーマをゼロから設計した。エンティティ抽出、正規化判断、履歴・監査の持ち方、命名規約、拡張性まで含む。
- **判断点**: 業務概念の写像精度 / 将来の変化耐性 / 性能・運用への配慮
- **見極め**: 「ER 図を引いただけ」ではなく、削除・履歴・soft delete・監査の方針を最初から設計しているか。命名と粒度の判断根拠を語れるか。
- **射程**: 機能〜プロダクト

### D-14: システム全体アーキテクチャをゼロから引く
- **状況**: 新規プロダクトや大規模刷新で、システム全体のアーキテクチャ（構成図、技術選定、データフロー、デプロイ構成）をゼロから設計し、合意形成・PoC まで持っていった。
- **判断点**: ビジネス要件の翻訳 / 技術選定の根拠 / 段階導入と最終形のバランス
- **見極め**: 構成図の各要素について「なぜ採用したか」「他に何を比較したか」を語れるか。「最初から完成形」ではなく段階展開の設計があるか。
- **射程**: プロダクト〜事業

### D-15: コードベース内部のレイヤー設計・ディレクトリ構成
- **状況**: アプリケーションのレイヤー設計（Clean / Layered / Hexagonal / DDD など）、ディレクトリ構造、パッケージ責務、DI 構造などを設計・改善した。
- **判断点**: 純粋性 vs 開発速度 / チームスキル分布との適合 / テスト容易性
- **見極め**: 流派を暗唱するのではなく、自プロダクトの文脈で「ここまでやる / ここはやらない」を判断できるか。失敗した抽象化の経験を語れるか。
- **射程**: 機能〜プロダクト

## 2. トラブル対応パターン（T, 12個）

中核は「**何が起きていて、どの仮説から検証したか、なぜその順序か**」。
障害対応の経験は、観測性・推論・組織連携の総合力を映す。

### T-01: 原因不明の本番障害の一次対応
- **状況**: 本番サービスで原因不明の障害が発生し、ログ・メトリクス・トレースから症状を切り分け、暫定対応・復旧判断を行った。
- **判断点**: 原因究明 vs 復旧優先 / 暫定対応の副作用評価
- **見極め**: 「最初に何を見たか、なぜそれを見たか」の論理。捨てた仮説と理由を語れるか。
- **射程**: 機能〜プロダクト

### T-02: メモリリーク／スレッドリークによる徐々の劣化
- **状況**: 一定時間運用後にレスポンスが劣化する／OOM で再起動するなど、徐々に進行する劣化型障害を特定・対処した。
- **判断点**: 一時しのぎ（再起動運用）vs 真因対処の優先度判断
- **見極め**: ヒープダンプ・スレッドダンプ・プロファイラの読み方。GC ログから何を読むか。
- **射程**: 機能〜プロダクト

### T-03: 外部依存先の障害による波及
- **状況**: 外部 API / SaaS / クラウドサービスの障害により、自社サービスが波及障害を起こした。
- **判断点**: 自社責任範囲 vs 依存先責任 / サーキットブレーカー導入や fallback の必要性
- **見極め**: 「依存先 SLA を超えた信頼性を自社が出せるか」の判断軸を持つか。
- **射程**: 機能〜プロダクト

### T-04: 部分的に遅い／失敗するという症状の絞り込み
- **状況**: 一部のユーザー / 一部のリクエスト / 特定時間帯だけ遅い・失敗するという症状から、再現条件を絞り込み原因に到達した。
- **判断点**: ログ採取の優先度 / ユーザー影響の最小化と並行調査
- **見極め**: 「平均では正常」に騙されないか。p99 / 特定経路の異常を捉える観測手法を持つか。
- **射程**: タスク〜機能

### T-05: 真因がインフラ層にあった障害
- **状況**: アプリケーション層で原因が見つからず、ネットワーク・DNS・証明書・IAM・k8s リソースなどに真因があった障害を解決した。
- **判断点**: アプリ層調査の見切り判断 / インフラ担当との切り分け連携
- **見極め**: tcpdump / dig / 証明書チェーン確認 / cgroup・OOM Killer など、インフラ層の素養。
- **射程**: 機能〜プロダクト

### T-06: 並行処理特有のバグ
- **状況**: 競合状態（race condition）、デッドロック、メモリ可視性、リオーダといった並行処理特有のバグを調査・修正した。
- **判断点**: 再現の困難さに対する戦略（ストレステスト、解析、見直し範囲）
- **見極め**: 「たまたま動いている」コードを見抜けるか。同期プリミティブの選択理由を持つか。
- **射程**: 機能

### T-07: データ不整合・破損の復旧
- **状況**: バグ・障害・運用ミスによりデータ不整合・破損が発生し、復旧と原因究明を並行で進めた。
- **判断点**: 復旧優先度 / 顧客影響開示 / バックアップ運用の前提が成立していたか
- **見極め**: ユーザー影響の特定方法、影響範囲調査の SQL / スクリプトを設計できるか。
- **射程**: プロダクト〜事業

### T-08: 障害時のステークホルダー対応
- **状況**: 障害発生時に経営・CS・他チーム・ユーザーに対し、状況・影響・復旧見込みを伝え、判断を仰いだ。
- **判断点**: 開示範囲・タイミング / 復旧確度の伝え方 / 補償判断
- **見極め**: 「技術的に正しい」と「組織として伝わる」のギャップに自覚があるか。
- **射程**: 事業〜組織

### T-09: ポストモーテムと再発防止
- **状況**: 障害後にポストモーテムを書き、再発防止策を実装まで持っていった。または、ポストモーテム文化自体の導入・改善を行った。
- **判断点**: 個人の責任追及 vs システム改善 / 投資コストとリスクの天秤
- **見極め**: 「人を責めない」を保ちつつ説明責任は果たすバランス感。
- **射程**: 機能〜組織

### T-10: 観測性の不足を認識した投資判断
- **状況**: ある障害をきっかけに、メトリクス・ログ・分散トレースが足りていないと判断し、観測性を上げる投資をした。
- **判断点**: 投資コスト vs 次の障害コスト / どこを観測するかの優先度
- **見極め**: SLI / SLO の設計、ログ設計の構造化、トレース ID 伝播の徹底など、設計の解像度。
- **射程**: 機能〜プロダクト

### T-11: 移行・リリース後の不具合と前進・撤退判断
- **状況**: 大きな移行・リリース後に不具合が顕在化し、ロールバック / 前進修正 / 部分有効化を判断した。
- **判断点**: ロールバックの可逆性 / ユーザー影響の見積もり / データ移行を伴う場合の不可逆性
- **見極め**: 「ロールバックできない移行」を作らないための事前設計を持っているか。
- **射程**: プロダクト〜事業

### T-12: 「再現できないバグ」の取り扱い
- **状況**: 限られたユーザーからの報告のみで、社内では再現できないバグに対応した。
- **判断点**: 限定ログ取得 / ヒアリング / A/B 出し分け / 静観の選択
- **見極め**: 「再現できないから対応できない」ではなく、限定有効化や追加観測でデータを取る発想。
- **射程**: 機能〜プロダクト

## 3. 性能・規模パターン（P, 8個）

中核は「**測ったか、見積もったか、根拠で動いたか**」。
カンと経験則ではなく、計測・モデル化に基づく判断ができるかを引き出す。

### P-01: 想定外のトラフィック増によるボトルネック特定
- **状況**: 急なアクセス増（キャンペーン、SNS バズ、流入経路追加など）で性能問題が顕在化し、ボトルネックを特定して対処した。
- **判断点**: アプリ / DB / NW / FE のどこから攻めるか / 暫定対処と恒久対処の分け方
- **見極め**: 「どこを最初に見たか、なぜ」を時系列で語れるか。捨てた仮説の理由。
- **射程**: 機能〜プロダクト

### P-02: クエリ性能最適化
- **状況**: クエリ性能（N+1、フルスキャン、JOIN 過多、不適切なインデックス）の問題を発見し、最適化した。
- **判断点**: クエリ書き換え vs インデックス追加 vs スキーマ変更 vs キャッシュ追加
- **見極め**: EXPLAIN / 実行計画の読み方、インデックス追加の副作用（書込性能、サイズ）を語れるか。
- **射程**: 機能

### P-03: キャッシュ戦略の設計と落とし穴
- **状況**: 多層キャッシュ（CDN / リバースプロキシ / アプリ内 / Redis / DB レベル）の設計や改善を行った。キャッシュ汚染、スタンピード、整合性の問題への対処を含む。
- **判断点**: 鮮度 vs 性能 / TTL 設計 / 無効化戦略
- **見極め**: 「キャッシュ整合性」の罠を経験で語れるか。stampede 対策（lock, jitter, refresh-ahead）を知っているか。
- **射程**: 機能〜プロダクト

### P-04: 大量データ処理の設計
- **状況**: バッチ・ETL・大規模エクスポート・集計など、大量データを扱う処理を設計し、性能・コスト・運用負荷を成立させた。
- **判断点**: 並列度 / メモリ vs ストレージ / 分散処理の導入
- **見極め**: バックプレッシャー、チェックポイント、再実行性、idempotency を含む設計ができるか。
- **射程**: 機能〜プロダクト

### P-05: スケーリング判断
- **状況**: 水平 / 垂直スケール、シャーディング、Read Replica、リーダー選出など、スケーリングの方式を判断・実装した。
- **判断点**: スケール限界の見積もり / 運用複雑性 / コスト
- **見極め**: ステートフル成分の取り扱い、シャードキーの選び方の根拠を語れるか。
- **射程**: プロダクト

### P-06: 負荷試験の設計と結果反映
- **状況**: 負荷試験を設計・実施し、結果から設計・実装を変更した。
- **判断点**: 想定シナリオの作り方 / どこをサチらせるか / 本番との乖離許容
- **見極め**: 「試験で OK だったが本番で問題」を経験から語れるか。リハーサル設計の解像度。
- **射程**: 機能〜プロダクト

### P-07: 容量計画と非機能要件
- **状況**: ピーク QPS、ストレージ増加、レイテンシ目標、可用性目標などを定義・見積もりし、設計に反映した。
- **判断点**: 過剰投資 vs 過少設計 / SLA / SLO の握り
- **見極め**: フェルミ推定で容量を見積もれるか。SLO とエラーバジェットの運用観念。
- **射程**: プロダクト〜事業

### P-08: 裾野レイテンシ（p99 / p999）の改善
- **状況**: 平均レイテンシは問題ないが、p99 / p999 で問題が出る、テールが太い問題を特定・改善した。
- **判断点**: GC 起因 / コンテンディション起因 / 外部依存起因 / cold start 起因の切り分け
- **見極め**: 「平均で見ない」感覚を持つか。ヘッジリクエスト・タイムアウト・並列実行などの引き出しがあるか。
- **射程**: 機能〜プロダクト

## 6. AI 活用・AI ネイティブ開発パターン（A, 6個）

中核は「**AI を道具として使う判断、AI を機能として組み込む判断、その両方を文脈で使い分けられるか**」。
LLM・RAG・エージェントの実装経験そのものに加え、決定論的処理との境界、評価・改善ループ、コスト・品質のトレードオフを引き出す。

> **このカテゴリの限界に関する留保**: AI ネイティブ領域は技術トレンドの変化が速く、創業者自身の暗黙知の蓄積もこれからの領域である。MVP 検証時はパターンの判別性そのものを評価対象とし、磨き込みの最優先候補として継続的に更新する前提で扱う。

### A-01: LLM をプロダクト機能に組み込む設計判断
- **状況**: 機能要件に対し、LLM をどう組み込むか（生成、分類、抽出、要約、対話など）を判断し、プロンプト構造・フォールバック・UX を設計した。
- **判断点**: LLM に任せる範囲 vs 決定論的処理 / 失敗時の UX / システムプロンプト設計
- **見極め**: 「とりあえず LLM」ではなく、なぜ LLM か・なぜ既存手法でないかを語れるか。失敗ケースのフォールバック設計を持つか。
- **射程**: 機能〜プロダクト

### A-02: RAG システムの設計判断
- **状況**: 社内ドキュメント・製品知識・大規模コンテンツに対し、RAG（Retrieval-Augmented Generation）を設計・実装した。チャンク戦略、embedding 選定、検索方式、リランク、評価を含む。
- **判断点**: チャンクサイズ・分割粒度 / ベクトル検索 vs ハイブリッド / 検索精度と生成品質のトレードオフ
- **見極め**: 「精度が出ない」原因を retrieval / generation / プロンプトのどこに切り分けられるか。評価データセットの作り方を持つか。
- **射程**: 機能〜プロダクト

### A-03: AI エージェント設計
- **状況**: 複数ステップのタスクを実行する AI エージェントを設計・実装した。ツール定義、ループ制御、ガードレール、コスト管理、人間介在点（HITL）を含む。
- **判断点**: 自律性の許容範囲 / 失敗時のリカバリ / 人間レビューの差し込みポイント
- **見極め**: 暴走・無限ループ・コスト爆発の経験談を持つか。ツール粒度と境界の設計判断を語れるか。
- **射程**: 機能〜プロダクト

### A-04: LLM API のコスト・レイテンシ・品質トレードオフ
- **状況**: 本番運用に向けて、LLM API のモデル選定、プロンプトキャッシュ、ストリーミング、バッチ処理、ファインチューニング、SLM 活用などを判断した。
- **判断点**: 大型モデル vs 小型モデル / API vs 自社ホスティング / リアルタイム vs バッチ
- **見極め**: コスト見積もりを QPS × トークン数で回せるか。スケール時の経済性を語れるか。
- **射程**: 機能〜プロダクト

### A-05: AI 機能の評価・改善ループ設計
- **状況**: AI 機能の品質を継続的に評価・改善する仕組みを設計した。オフライン評価データセット、A/B テスト、人手評価、ハルシネーション検出、回帰テストを含む。
- **判断点**: 何を「品質」と定義するか / 自動評価 vs 人手評価 / 評価コスト
- **見極め**: 「LLM as judge」の限界を理解しているか。golden set のメンテナンス運用を語れるか。
- **射程**: 機能〜プロダクト

### A-06: AI ネイティブプロダクトの責務設計
- **状況**: AI を中心機能として据えるプロダクトで、決定論的処理と LLM の責務分担、検証可能性、ユーザーへの説明性、エラー時の責任設計を判断した。
- **判断点**: ブラックボックスの許容範囲 / 監査ログ設計 / ユーザーが結果を信用できる設計
- **見極め**: 「AI が間違えた時、誰がどう気づくか」を運用視点で語れるか。説明可能性と性能のトレードオフを認識しているか。
- **射程**: プロダクト〜事業

## 8. Week 2 への引き継ぎ：深掘り 4 段階の入口

各パターンに対し、Week 2 で以下の 4 段階の深掘り設計を行う：

- **第 1 段（経験有無）**: 「この状況に直面したことがあるか」
- **第 2 段（経験の真贋）**: 「具体的にどんな症状で、どの順序で何を見たか」
- **第 3 段（判断力）**: 「どう解決したか、なぜそれを選んだか、捨てた選択肢は」
- **第 4 段（メタ認知）**: 「規模が違えば同じ判断をするか、今ならどう変えるか」

Week 2 では、各パターンに対し以下を設計する：

1. **第 1 段の問い掛け文（候補者の自己申告を引き出す導入）**
2. **第 2 段の深掘り質問テンプレート（症状・経過・選択肢の確認）**
3. **第 3 段の深掘り質問テンプレート（判断根拠と代替案）**
4. **第 4 段のメタ認知質問テンプレート（射程拡張と再判断）**
5. **見極め基準（経験者の典型回答と非経験者の限界点）**
6. **AI 活用観点の差し込み（このパターンで「AI に任せた / 任せなかった」境界を聞く）**
