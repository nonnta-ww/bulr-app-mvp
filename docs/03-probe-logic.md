# bulr 問診設計：深掘り質問ロジック

このドキュメントは、`docs/02-questionnaire-patterns.md` で定義した 57 個の状況パターンに対し、**面接アシスタント型** の bulr が、面接中に面接官へ深掘り質問を提案するための **4 段階質問ロジックの設計** をまとめる。

参考：`docs/bulr-handoff.md`（戦略全体・v3 マージ済み）, `docs/bulr-product-direction.md`（プロダクト方向性）, `docs/bulr-prd-year1.md`（Year 1 PRD）, `docs/02-questionnaire-patterns.md`（状況パターン集）

---

## 0. 設計哲学と前提

### 0.1 何を作っているのか

「**人間の面接官が候補者と 30〜40 分対話する中で、LLM が面接官に次の深掘り質問を提案し、回答を構造化して可視化する**」ための問いかけロジックを作っている。

bulr は AI 面接官ではなく、**面接官の隣に座る面接アシスタント**。

- 面接官は人間。候補者との対話の主役。
- LLM は黒子。候補者の回答を分析し、面接官に深掘り質問の選択肢を提示する。
- 最終判断は常に面接官にある。LLM の提案を採用するか、自分の判断で別の質問を投げるかは面接官が決める。

このドキュメントは、面接アシスタント型 bulr の中核ロジックを定義する。同時に、共通の問診ロジックの上で対話の主体だけを切り替えれば候補者直接対話型にも展開できるよう、抽象度を保つ。

### 0.2 LLM と面接官の責務分担

| 領域 | LLM の責務 | 面接官の責務 |
|---|---|---|
| 対話の進行 | 質問候補の提案、評価サマリーの生成 | 質問の選択、対話の実施、空気感の判断 |
| 回答の解釈 | 回答の構造化、第何段まで到達したかの評価 | 回答の補完的解釈、現場感での真贋判定 |
| 詰まり判定 | 抽象的な答え・矛盾・数値欠落を検出して提案 | 検出された詰まりを踏まえて次の動きを決定 |
| パターン遷移 | 「次のパターンに進む」候補の提示 | パターン移行のタイミング決定 |
| 時間配分 | 進捗の可視化、残時間の表示 | 最終的な時間管理 |
| 記録 | 回答の構造化保存、評価スコアの記録 | 評価メモの記入、最終判断 |

LLM は決定者ではなく**助言者**。面接官の判断を支援することが目的。

### 0.3 4 段階の意図

| 段 | 何を測る | 質問の性格 | 通過の兆候 | 詰まりの兆候 |
|---|---|---|---|---|
| 第 1 段 | 経験の有無 | 状況に直面したことがあるか | 状況を具体的に再現できる（固有名詞、時期、規模） | 抽象的な肯定、一般論への退避 |
| 第 2 段 | 経験の真贋 | どんな症状で、どう特定したか | 一貫した時系列、捨てた仮説、関係者の役割 | 時系列の破綻、数値の欠落、当事者の不在 |
| 第 3 段 | 判断力 | どう解決し、なぜそれを選んだか | 複数選択肢の比較、トレードオフの言語化、後悔の自覚 | 「正解」を語る、選択肢が単一、後悔がない |
| 第 4 段 | メタ認知 | 規模・文脈が違えば同じ判断をするか | 文脈差異への感度、自分の判断のバイアス自覚 | 「今でも同じ」と言い切る、規模感への鈍感 |

### 0.4 セッション全体の流れ（30〜40 分の使い方）

```
[ 面接前 ]
   - 面接官が bulr の「パターン選択画面」を開く
   - 候補者情報を入力（名前、応募職種、経験年数、自由メモ、経歴ファイル参照）
   - 57 パターンから「今日深掘りするもの」を 10〜12 個選ぶ
   - （Stage 2 では事前問診結果から AI が推薦、面接官が承認・調整）

[ 0〜5 分 ] イントロ
   - 候補者の経歴を簡単に確認
   - 「今日はいくつかの状況について深くお聞きします」と趣旨説明
   - 録音同意の確認

[ 5〜35 分 ] 各パターンの第 1〜4 段
   - 1 パターンあたり 3〜5 分
   - 「状態 A：質問表示 + 録音中」→「状態 B：LLM 分析 + 3 候補提示」のループ
   - 各パターンを第 1 段 → 第 2 段 → 第 3 段 → 第 4 段 と進める
   - 詰まったら無理に引き出さず、LLM が「次のパターンに進む」候補を提示
   - 面接官が判断してパターン遷移

[35〜40 分 ] クロージング
   - AI 活用観点の総括問い（横断軸）
   - 候補者からの質疑応答時間
```

### 0.5 「経験の濃淡」を測ることが目的

評価軸は「正解か / 不正解か」ではない。
**どの状況をどの深さまで経験しているか** をヒートマップ化することがゴール。
特定パターンで詰まることそのものが情報であり、ペナルティではない。

## 1. 評価ルーブリック（見極め基準の体系）

### 1.1 真贋スコア（経験が本物か）

| 段階 | 観察項目 | 経験者の典型 | 非経験者・伝聞・盛りの典型 |
|---|---|---|---|
| 具体性 | 固有名詞、数値、時期 | 「2022 年に〜のサービスで、QPS が 800 から 3000 に」 | 「ある程度の規模で」「結構な負荷で」 |
| 時系列 | 出来事の前後関係 | 検知 → 仮説 → 検証 → 対処 が一貫 | 順序が後付けで矛盾、検証ステップ抜け |
| 関係者 | 当事者と役割 | 「DBA の〇〇さんと相談、PdM が判断」 | 「チームで」「上の人と相談」 |
| 捨てた仮説 | 検討して棄却した選択肢 | 「最初は〜を疑ったが、〜の理由で違うと判断」 | 一直線に正解にたどり着く語り |
| 後悔 | 振り返って改善したい点 | 「今思えば〜すべきだった」 | 「ベストを尽くした」「問題なかった」 |

### 1.2 判断力スコア（トレードオフを認識しているか）

| 観察項目 | 高 | 低 |
|---|---|---|
| 選択肢の数 | 3 つ以上の現実的代替案を比較 | 1 つの選択肢だけ語る |
| トレードオフ言語化 | 「A は速いが B は柔軟」のように軸を出す | 「A が良かったから」 |
| 定量と定性 | 数値根拠と組織根拠の両方 | 片方のみ |
| 制約への感度 | 当時の制約（時間・人・スキル）を認識 | 制約を語らず理想論 |

### 1.3 射程スコア（判断のスケール）

| 射程 | シグナル |
|---|---|
| タスク | 自分の手元の処理を最適化した |
| 機能 | 1 つの機能の設計判断を主導 |
| プロダクト | プロダクト全体に影響する選択を判断 |
| 事業 | 事業の収益・契約・コンプラに影響する判断 |
| 組織 | チーム編成・採用・文化・プロセスに影響する判断 |

### 1.4 メタ認知スコア（文脈変化への感度）

| 観察項目 | 高 | 低 |
|---|---|---|
| 規模感度 | 「10 人規模ならやらない、100 人なら必要」と語れる | 規模に関わらず同じ判断と言う |
| フェーズ感度 | 「PMF 前 / 後で違う」「レガシー保守と新規で違う」 | フェーズの違いを感じない |
| バイアス自覚 | 「自分は〜寄りの判断をしがち」 | 自分の傾向を客観視できない |
| AI 影響感度 | 「2024 年以降なら AI を使って違う判断ができる」 | AI の登場を判断軸に入れない |

## 2. 段階別 共通質問バンク（再利用テンプレ）

LLM はパターンに応じて以下のテンプレを文脈化し、**面接官への提案質問** として組み立てる。

これらは面接官が候補者にそのまま聞ける質問文として提示される。面接官は提案を採用するか、自分の言葉で言い換えるか、別の質問を投げるかを判断する。

### 2.1 第 1 段 共通テンプレ

- 「〜という状況に直面したことはありますか？」
- 「もし経験があれば、どんなシステム・規模だったか一言で教えてください」
- 「直接担当しましたか、近くで見ていましたか、伝聞ですか」

### 2.2 第 2 段 共通テンプレ（症状・経過・選択肢の事実確認）

**症状の確認**
- 「最初に何を見て『これは問題だ』と気づきましたか？」
- 「症状を具体的に教えてください（メトリクス、ログ、ユーザー報告）」
- 「気づいた時点では、何が原因だと最初に疑いましたか？」

**経過の確認**
- 「気づいてから、どの順序で何を確認しましたか？」
- 「途中で『これは違う』と捨てた仮説はありますか？それはなぜ違うと判断したのですか？」
- 「どのくらいの時間がかかりましたか？」

**当事者の確認**
- 「あなた自身は何を担当しましたか？他に誰が関わりましたか？」
- 「決定権を持っていたのは誰ですか？」
- 「あなたが最も貢献したと思う部分はどこですか？」

### 2.3 第 3 段 共通テンプレ（判断根拠と代替案）

**選択肢の網羅**
- 「他に考えた選択肢はありますか？」
- 「最終案以外で『これも有りだった』と思うものは？」
- 「『これは絶対やらない』と最初から外した選択肢はありますか？」

**判断軸の言語化**
- 「最終的にその選択を選んだ決め手は何でしたか？」
- 「判断を分けた軸を 2〜3 個挙げるとしたら？」
- 「制約（時間・人・予算・スキル）が違えば、別の選択肢になりましたか？」

**コスト・影響評価**
- 「その判断のコストは何でしたか？（金銭、人時、運用負荷、技術負債）」
- 「想定していなかった副作用はありましたか？」
- 「効果は何で測りましたか？想定通りでしたか？」

### 2.4 第 4 段 共通テンプレ（メタ認知・射程拡張）

**規模の変化**
- 「もしチームが 10 倍だったら、同じ判断をしますか？」
- 「データ量が 100 倍なら、どこから違ってきますか？」
- 「ユーザー数が逆に小さい（一桁人）なら、何を省きますか？」

**フェーズ・文脈の変化**
- 「PMF 前のスタートアップなら、同じ判断ですか？」
- 「規制業界（医療・金融）なら、どこを変えますか？」
- 「プロダクトの重要度が違っていたら？」

**時間軸での再評価**
- 「今もう一度やるとしたら、何を変えますか？」
- 「当時の自分にアドバイスするなら、何を伝えますか？」
- 「この経験から得た、別の場面でも使う判断軸はありますか？」

**AI 活用観点（横断軸）**
- 「このプロセスで AI を使えるとしたら、どこを任せて、どこを自分でやりますか？」
- 「AI ツールが使える今、同じ問題に対するアプローチは変わりますか？」
- 「逆に『絶対 AI には任せない』部分はどこですか、なぜ？」

## 3. カテゴリ別 深掘り観点

### 3.1 D（設計判断）の深掘りで必ず聞くこと

- **比較した選択肢**: 「他に〜という選択肢はありましたか」
- **後悔した判断**: 「過剰設計 / 過少設計、どちらに偏りましたか」
- **撤退条件**: 「『この設計は間違いだった』と判定する基準を当時持っていましたか」
- **設計レビュー**: 「誰にレビューしてもらいましたか、反対意見はありましたか」

### 3.2 T（トラブル対応）の深掘りで必ず聞くこと

- **時系列**: 検知 → 仮説 → 検証 → 暫定対処 → 恒久対処の各時点
- **観測手段**: 「ログ、メトリクス、トレース、ヒアリング、再現」のうち何を使ったか
- **捨てた仮説**: 「最初は何を疑い、なぜ違うと判断したか」
- **再発防止**: 「同じことが起きない仕組みは入れましたか」

### 3.3 P（性能・規模）の深掘りで必ず聞くこと

- **数値**: QPS、レイテンシ（平均と p99）、データ量、ユーザー数
- **計測手法**: プロファイラ、APM、ログ集計、負荷試験のいずれか
- **見積もり vs 実測**: 「想定と実測のズレはありましたか」
- **コスト**: 「インフラコスト、開発コスト、運用負荷の比較」

### 3.4 S（セキュリティ・データ）の深掘りで必ず聞くこと

- **影響範囲調査**: 「誰が / 何件 / いつから影響があったか、どう特定したか」
- **開示判断**: 「誰がどのタイミングで知ったか、ユーザー通知はしたか」
- **コンプラ折衝**: 「法務・経営・監査との関与」
- **残留リスク**: 「直さなかったリスクと、それを許容した理由」

### 3.5 O（組織・プロセス）の深掘りで必ず聞くこと

- **反対意見の取り込み**: 「賛成しなかった人は誰で、何が論点でしたか」
- **撤退条件**: 「うまくいかなかったらどうするつもりでしたか」
- **効果計測**: 「導入効果は何で測りましたか、成功 / 失敗の判定基準」
- **失敗経験**: 「合意形成 / プロセス導入で失敗した経験」

### 3.6 A（AI 活用・AI ネイティブ開発）の深掘りで必ず聞くこと

- **なぜ AI か**: 「決定論的処理ではダメだった理由」
- **失敗モードの認識**: 「どんな失敗が起きうるか、観測できるか」
- **評価方法**: 「品質をどう測ったか、誰が評価者だったか」
- **コスト・レイテンシ**: 「QPS × トークン数で月額いくらだったか」
- **責任設計**: 「AI が間違えた時、誰がどう気づき、どう対処するか」

## 4. パターン別 核心質問（全 57）

各パターンに対し、以下のフォーマットで核心質問を定義する。

```
### [ID] [パターン名]
- **1段導入**: 経験有無を引き出す問いかけ文
- **2段の核**: 真贋を見極める焦点
- **3段の核**: 判断根拠を引き出す焦点
- **4段の核**: メタ認知を引き出す焦点
- **真贋シグナル**: 経験者なら出てくる固有のキーワード・数値・経験談
- **AI 観点**: このパターン固有の AI 活用問い
```

---

### D-01: モノリス内の機能の独立サービス化判断
- **1段導入**: 「モノリスの中の特定機能だけがスケール要件で突出し、独立サービスに切り出すか同居で吸収するかを判断した経験はありますか」
- **2段の核**: 何の機能か / どのスケール要件（QPS, データ量, バッチ時間）か / 結果は分割 or 吸収か / 移行期間と段階
- **3段の核**: 分割コスト評価（運用、観測、デプロイ）と吸収コスト評価の比較 / 反対意見と撤退条件
- **4段の核**: チーム規模が半分なら / 10 倍トラフィックなら / 今やり直すなら何を変えるか
- **真贋シグナル**: 境界引きの試行錯誤、「分割しすぎた / しなさすぎた」の後悔、ストラングラーパターンや Anti-Corruption Layer などの言及
- **AI 観点**: 設計レビューや影響分析で AI を使うとしたら何を任せるか

### D-02: 既存スキーマで窮屈になった新要件の取り込み
- **1段導入**: 「既存の RDB スキーマに収めようとすると無理が出る新要件で、テーブル分割・非正規化・別ストア導入を判断した経験はありますか」
- **2段の核**: どんな新要件か / どこが窮屈だったか / どの選択肢を採用したか / 移行手順
- **3段の核**: 「正規化原則」と「現実トレードオフ」の折り合い方 / 別ストア導入時のデュアルライト戦略
- **4段の核**: スタートアップの初期なら / レガシーで触れない部分が多ければ / 今ならどう判断
- **真贋シグナル**: スキーマ移行の苦しみ、整合性ウィンドウの設計、後悔の具体例
- **AI 観点**: スキーマ設計レビューや差分検出に AI を使う発想があるか

### D-03: 同期処理の非同期化・疎結合化
- **1段導入**: 「外部 API や重い処理に同期で依存していた箇所を、キュー・イベント駆動・キャッシュなどで疎結合化した経験はありますか」
- **2段の核**: 何が同期で何が問題だったか / どの方式で疎結合化したか / 整合性ウィンドウ
- **3段の核**: 同期の単純さを捨てる根拠 / 結果整合で良い領域の見極め / 失敗時の補償設計
- **4段の核**: 整合性が厳しい業務（決済等）なら / リアルタイム性要求が高ければ
- **真贋シグナル**: 重複処理対策、デッドレターキュー、idempotency キーの言及
- **AI 観点**: 影響分析や test 設計に AI を使う発想

### D-04: API スタイル選択と後方互換性戦略
- **1段導入**: 「新規 API または既存刷新で、REST / gRPC / GraphQL / イベントベース のいずれを選ぶか判断した経験はありますか」
- **2段の核**: 候補と選定基準 / 後方互換性の保ち方 / バージョニング方式
- **3段の核**: 「自分が好きな方式」ではなく文脈で選んだ根拠 / 互換破壊時の移行戦略
- **4段の核**: クライアントが社内のみなら / 外部公開なら / 多言語クライアントなら
- **真贋シグナル**: deprecation 期間の設計、SDK 配布の運用、Schema Registry の言及
- **AI 観点**: API 設計の自動化や OpenAPI からのコード生成への AI 活用

### D-05: 認証・認可方式の選定／置き換え
- **1段導入**: 「既存の認証方式が要件に合わなくなり、IdP 導入・SSO・トークン方式置き換えなどを判断した経験はありますか」
- **2段の核**: 何が問題で何に変えたか / ユーザー影響の抑え方 / 移行期間の併存設計
- **3段の核**: 自前運用 vs マネージド / セキュリティ vs UX / 既存トークンの扱い
- **4段の核**: BtoC / BtoB で変わるか / 規模が小さければ自前で十分か
- **真贋シグナル**: トークン失効の設計、リフレッシュ戦略、SCIM や JIT プロビジョニング
- **AI 観点**: 不審アクセス検知への ML 利用の検討

### D-06: トランザクション境界と整合性モデル
- **1段導入**: 「複数リソースに跨る処理で、ACID トランザクション vs 結果整合（Saga, Outbox）を判断した経験はありますか」

- **1段導入**: 「AI を中心機能として据えるプロダクトで、決定論的処理と LLM の責務分担、検証可能性、ユーザーへの説明性、エラー時の責任設計を判断した経験はありますか」
- **2段の核**: 責務分担 / 監査ログ設計 / ユーザーへの説明方法
- **3段の核**: ブラックボックスの許容範囲 / 「AI が間違えた時、誰がどう気づくか」の運用設計
- **4段の核**: BtoB エンタープライズなら / 規制業界なら / 法的責任が明確に求められれば
- **真贋シグナル**: 「人間の最終承認」設計、信頼度スコアの提示、AI 出力のバージョニング
- **AI 観点**: AI を使う側の倫理・責任観そのものが問い

## 5. AI 活用観点（横断軸）の差し込み方

### 5.1 横断軸の意図

ハンドオフ文書で定義した「**AI 時代における中堅以上エンジニアの実務判断力**」の核心軸。
全パターンに対し、第 4 段の最後に **AI 活用観点の問い** を LLM が提案候補として差し込む。
**「AI を道具として使いこなす判断ができるか」「AI に任せない判断ができるか」** を面接官が引き出せるようにする。

### 5.2 差し込みの基本パターン

各パターンの第 4 段に近づいた段階で、LLM は以下のいずれかを提案候補に含める：

- 「このプロセスで AI を使えるとしたら、どこを任せて、どこを自分でやりますか？」
- 「AI ツールが使える今、同じ問題に対するアプローチは変わりますか？」
- 「逆に『絶対 AI には任せない』部分はどこですか、なぜ？」

### 5.3 セッション末の総括問い

セッションの最後（35〜40 分）の段階で、LLM は以下を提案候補に含める：

- 「今回話したいくつかの状況のうち、AI を使えば違う判断・解決ができたものはありますか」
- 「あなたが AI に任せたくない・任せられない判断は何ですか」
- 「これから AI 前提で開発するチームを作るなら、何を一番先に変えますか」

### 5.4 評価ルーブリック（AI 活用観点）

| スコア | 兆候 |
|---|---|
| 高 | AI を「使う / 使わない」の境界を文脈で判断できる、失敗モードを認識している、自分のチームでの導入経験が具体的 |
| 中 | AI を使った経験はあるが境界が曖昧、AI で何が変わるかは語れる |
| 低 | AI を使ったことがない、または「使えば全部解決」「絶対使わない」の極端な立場 |

## 6. LLM プロンプト設計指針

### 6.1 LLM 面接アシスタントの責務

LLM は面接の進行を支配しない。**面接官への提案者**として動作する。

役割：

1. **回答の構造化**: 候補者の発話を文字起こしから抽出し、第 1〜4 段のどこに位置するかを判定
2. **評価スコアの算出**: 1 章の評価ルーブリックに基づき、真贋・判断力・射程・メタ認知・AI 活用の各軸を採点
3. **次の質問候補の提案**: 共通テンプレ（2 章）、カテゴリ別観点（3 章）、パターン別核心質問（4 章）から、3 つの質問候補を生成
4. **詰まり検出**: 抽象的な答え・矛盾・数値欠落を検出し、面接官に「次のパターンに進む」候補を提示
5. **進捗の可視化**: 現在のパターン進捗、残時間、カバー済みパターン数を表示

### 6.2 LLM の出力フォーマット

各「次の質問へ」ボタン押下時に、LLM は以下を生成する。**ターン単位の評価は定性ラベル**（`observed`／`partial`／`absent`）で行い、**数値スコアへの集約はパターン完了時に `aggregatePatternCoverage` が担う**（rubric の `LlmTurnAnalysis` / `LlmEvaluation` 設計に準拠）。

```yaml
turn_analysis:
  pattern_id: "P-04"
  current_stage: 2  # 1, 2, 3, 4 のいずれか
  candidate_response_summary: |
    候補者は ECサイトの商品一覧で N+1 が発生し、
    Datadog APM で特定したと述べた。
  evaluation:                       # per-turn は定性ラベル (LlmTurnAnalysis)
    authenticity: "observed"        # observed / partial / absent
    judgment: null                  # 第 2 段では未評価
    meta_cognition: null
    ai_literacy: null
    scope_signal: null              # 1 | 2 | 3 | 4 | 5 | null
  authenticity_signals_observed:
    - "具体的サービス（ECサイトの商品一覧）"
    - "具体的ツール（Datadog APM）"
  authenticity_signals_missing:
    - "数値（クエリ数、レイテンシ）が未確認"

next_question_candidates:
  - text: "特定後、どう解決しましたか? 検討した選択肢も含めて"
    rationale: "第 3 段へ進む。判断と代替案を確認"
    intent: "deep_dive"             # spec: question_intent enum
    target_stage: 3
  - text: "規模が 10 倍だった場合、同じ解決方法を取りますか?"
    rationale: "第 4 段へ進む。メタ認知の確認"
    intent: "meta_cognition"
    target_stage: 4
  - text: "次のパターン（D-07: バッチ vs ストリーミング）に進む"
    rationale: "P-04 は十分カバー。次の領域へ"
    intent: "next_pattern"
    target_stage: null

session_progress:
  patterns_covered: 7
  patterns_remaining: 5
  elapsed_minutes: 23
  remaining_minutes: 17
```

パターン完了時には `aggregatePatternCoverage` が複数ターンの `evaluation` を集約し、`pattern_coverage.llm_evaluation` に **数値スコア** を書き出す：

| 軸 | per-turn の形 | 集約後（pattern_coverage.llm_evaluation） |
|---|---|---|
| authenticity | `observed/partial/absent` | `0-3 整数` （0 = 該当なし／詰まり） |
| judgment | 同上 | `0-3 整数` |
| meta_cognition | 同上 | `0-3 整数` |
| ai_literacy | 同上 | `0-3 整数` |
| scope | `scope_signal: 1-5 \| null` | `1-5 整数`（観察された最大射程を採用） |

このフォーマットは UI（状態 B）でそのまま表示できる構造にしておく。

### 6.3 矛盾検知のヒューリスティクス

- **時系列の破綻**: 「最初に〜と判断した」が後の発言と矛盾
- **規模の不一致**: 「大規模システム」と言いながら数値が小さい
- **当事者の不在**: 「私が判断した」と言いながら判断軸を語れない
- **後悔の欠落**: 「ベストを尽くした」「問題なかった」だけで自己批判が出ない

矛盾を検知した場合、LLM は「矛盾を詰問する質問」ではなく「**別の角度から確認する質問**」を提案候補に含める（例：「もう少し具体的な数値で言うと?」「他のメンバーはどう関わっていましたか?」）。

最終的に矛盾を踏まえて判断するのは面接官。LLM は判断の材料を提示する。

### 6.4 提案質問の言語的工夫

LLM が生成する質問候補は、面接官がそのまま読み上げても自然な日本語であること。

- **オープンクエスチョン優先**: 「はい / いいえ」だけで終わらせない
- **続きを促す**: 「もう少し詳しく」「他には?」
- **簡潔さ**: 1 つの質問につき 1 つの問い、長文で複数の問いを混ぜない
- **詰問にならない**: 「なぜ?」を連発しない、「教えてください」「聞かせてください」を使う
- **詰まり時の救済**: 「経験がなくても問題ありません」を含める提案も用意する

### 6.5 「自分で次を聞く」への対応

面接官が状態 B で「自分で次を聞く」を選択した場合、LLM は：

1. 次のターンの録音を待つ
2. 録音から面接官の発話（質問）と候補者の発話（回答）を分離
3. 面接官の質問を「投げられた質問」として記録
4. 候補者の回答を通常通り分析し、評価ロジックに渡す

これにより、LLM の提案を採用しない場合でも、データの構造化は維持される。

### 6.6 終了判定ロジック

LLM は以下の状態を検出した時、面接官に「次のパターンへ進む」候補を提示する：

- 経験あり：第 1〜4 段を一通り通過 → 次パターンへの遷移候補を提案
- 第 2 段で詰まり：真贋スコア低と記録 → 次パターンへの遷移候補を提案
- 第 3 段で詰まり：判断力スコア低と記録 → 第 4 段は省略して次パターンへの遷移候補を提案
- 経験なし：第 1 段で正直に「経験なし」と答えた場合、ペナルティなしで次へ進む候補を提案

最終的に次パターンへ進むタイミングは面接官が決定する。

### 6.7 出力データ構造（実装への橋渡し）

実装スキーマは `assessment-engine` spec が所有する。本節は spec の 5 テーブル構成（`interview_session` / `question_proposal` / `interview_turn` / `pattern_coverage` / `session_report`）を、本問診設計の観点から要約する。カラム名・enum 値は spec の DDL に揃える。

```
interview_session  (面接 1 件 = 1 行)
├ id
├ interviewer_id          (FK → user。面接官)
├ candidate_id            (FK → candidate)
├ status                  (draft / in_progress / completed / abandoned)
├ role                    (e.g. 'backend'。Stage 1 は backend のみ)
├ planned_pattern_codes[] (面接前に選択した 10〜12 個のパターンコード)
├ consent_obtained_at, consent_version
├ started_at, completed_at
└ created_at, updated_at

question_proposal[]  (1 ターン分の 3 候補を 1 行で表現)
├ id
├ session_id              (FK → interview_session)
├ prepared_for_turn_no    (この提案が対応する interview_turn の sequence_no)
├ candidate_1_text, candidate_1_intent  (deep_dive / meta_cognition / next_pattern)
├ candidate_2_text, candidate_2_intent
├ candidate_3_text, candidate_3_intent
├ selected_index          (1 / 2 / 3 / null。null = 面接官が「自分で次を聞く」を選択)
└ generated_at

interview_turn[]  (1 ターン = 1 行)
├ id
├ session_id              (FK → interview_session)
├ sequence_no             (面接内でのターン番号)
├ pattern_id              (FK → assessment_pattern、nullable。フリー質問は null)
├ proposal_id             (FK → question_proposal、nullable)
├ question_source         (llm_candidate_1 / llm_candidate_2 / llm_candidate_3 / manual)
├ question_text           (実際に投げられた質問)
├ audio_key               (Vercel Blob のキー、30 日後 NULL 化)
├ audio_expires_at        (削除予定時刻)
├ transcript jsonb        ({ interviewer?: string, candidate: string, raw: string })
├ llm_analysis jsonb      (6.2 の turn_analysis 構造。pattern_id, current_stage, evaluation, signals)
├ pattern_match_confidence(exact / inferred_high / inferred_low / off_pattern)
├ off_pattern_summary     (off_pattern の場合のみ、フリー質問の要約)
├ duration_ms
└ created_at

pattern_coverage[]  (1 セッション × 1 パターン = 1 行、UNIQUE(session_id, pattern_id))
├ id
├ session_id, pattern_id  (FK)
├ level_reached           (0-4。0 = 経験なし、1-4 = 第 1〜4 段到達)
├ stuck_type              (not_experienced / shallow / single_option / rigid、nullable)
├ llm_evaluation jsonb    (5 次元スコア: authenticity 0-3, judgment 0-3, scope 1-5,
│                          meta_cognition 0-3, ai_literacy 0-3, notes)
├ manual_evaluation jsonb (面接官の手動評価、admin-review-panel で記入。nullable)
├ turn_ids[]              (該当ターンの ID 配列)
└ finalized_at

session_report  (1 セッション = 1 行、UNIQUE(session_id))
├ id
├ session_id              (FK)
├ heatmap_data jsonb      (カテゴリ別カバレッジ + 5 次元別集約)
├ summary_text            (5 次元別所感 + カテゴリ別カバレッジ + フリー質問総評)
└ generated_at
```

スコアレンジの正規定義は `steering/evaluation-rubric.md` を参照。射程のみ 1-5、残り 4 軸は 0-3 整数。フリー質問は `interview_turn.pattern_id=null` + `pattern_match_confidence='off_pattern'` で記録し、`pattern_coverage` には集約しない（`off_pattern_summary` を `session_report.summary_text` の総評に反映）。

### 6.8 Stage 1 検証で重要なデータ

以下のデータは、Stage 1 で「対話型問診が実力を見抜けるか」を検証するために特に重要：

- **LLM 提案質問の採用率**: `interview_turn.question_source` の分布（`llm_candidate_1/2/3` vs `manual`）。3 候補のうちどれを選びやすいか、`manual` 率はどれくらいか
- **LLM 評価と面接官評価の一致度**: `pattern_coverage.llm_evaluation` と `pattern_coverage.manual_evaluation` の 5 次元スコア相関（admin-review-panel で集計）
- **詰まり判定の精度**: `question_proposal` のいずれかの候補が `intent='next_pattern'` だった時、面接官がそれを採用した率（`selected_index` で判定）
- **面接時間の最適化**: 1 パターンあたりの平均 `duration_ms` 合計、`pattern_coverage` 件数の完走率、`session_status='completed'` の完了率

## 7. レビュー観点（このドキュメントを読む創業者へ）

1. **段階間の繋がり**: 第 1〜4 段の問いが、ある状況の経験者には淀みなく続き、非経験者には早めに詰まる設計になっているか
2. **共通テンプレの汎用性**: 2 章のテンプレが多くのパターンで使い回せるか
3. **カテゴリ別観点の必要十分性**: 3 章の観点で各カテゴリの判別性が確保できるか
4. **核心質問の妥当性**: 4 章の「真贋シグナル」が、創業者が候補者と話す中で実際に聞き分けているサインと一致しているか
5. **AI 活用観点の差し込み方**: 5 章のやり方が「自然な対話の中で問える」レベルか、無理に挟むことになっていないか
6. **欠けている深掘り軸**: 創業者が普段見ている軸で、このルーブリックに含まれていないものはあるか
7. **LLM と面接官の責務分担**: 0.2 で定義した分担が、面接官の認知負荷を最小化できているか
8. **提案質問の自然さ**: 6.4 の言語的工夫が、実際の面接で違和感なく使えるか

特に **4（核心質問の妥当性）** は外からは検証できない。創業者の年間 800 書類のレビュー知見と、過去の面接で「こいつは経験者」「これは盛っている」と判別したサインを、このドキュメントの「真贋シグナル」と突き合わせて磨き込む必要がある。

また **7-8（面接アシスタント型固有の論点）** は実装後の検証で明らかになる。Stage 1 のベータで面接官のフィードバックを集めて磨き込む。

## 8. 実装への引き継ぎ

プロトタイプ実装で扱う内容：

1. **LLM プロンプトの実装**: 6.1〜6.6 を実際のシステムプロンプトに落とす
2. **データモデル**: 6.7 をスキーマに落とす（受験者、面接セッション、ターン記録、パターン応答、評価スコア）
3. **対話フロー**: 「事前パターン選択」→「状態 A」→「状態 B」→「状態 A」のループの状態機械
4. **音声処理**: Whisper API による文字起こし、話者分離プロンプトの実装
5. **観測・分析**: ターン記録ログから問診の妥当性を後から評価できる仕組み
6. **評価集約**: パターン応答 → カテゴリ別スコア → ヒートマップ集約のロジック（ただし MVP では UI は最低限）

## 9. 想定される拡張

- **パターン追加・改廃の運用**: 検証から得た学びを反映するための、パターンの追加・統合・削除の運用設計
- **職種別テンプレ**: フロントエンド / SRE / PdM 領域での 4 段階テンプレ（Stage 2 で追加）
- **AI 領域の磨き込み**: A カテゴリは技術トレンドの変化が速いため、3 ヶ月ごとに見直し
- **事前問診との統合**: Stage 2 で導入する事前問診の選択式回答から、AI が推薦パターンを自動算出
- **多言語対応**: 英語、ベトナム語、その他言語への翻訳と現地ニュアンスの調整（Stage 3 以降）
- **比較評価の自動化**: 同一パターンに対する複数候補者の応答を比較・ランキングする仕組み
- **面接官較正機能**: Stage 2 の中核機能。面接官ごとの評価傾向の偏りを可視化し、組織全体の採用基準を統一
- **候補者直接対話型への拡張**: Stage 3 以降のオプション。同じ問診ロジックを LLM が候補者と直接対話する形に展開
