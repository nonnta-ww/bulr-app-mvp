# Design Document — teamwork-style-diagnosis（チームワーク・スタイル診断）

- 日付: 2026-07-16
- ステータス: ブレインストーミング合意済み（spec 化前のデザインブリーフ）
- 種別: 候補者向け診断ファミリー 3つ目の原子診断 ＋ 面接官向け問診パターン（派生 spec）

## 背景・課題

「EQ / SQ のような観点で"社会性"を測り、入社後にメンバーとハレーションが起きないかチェックしたい」という採用現場のニーズが出発点。ただし相談内容には性質の異なる2目標が混在していたため、以下のように切り分けた。

- **A. 社会性特性の可視化**（本人の対人・協働スタイル）— 実現可能。今回作る。
- **B. 入社後ハレーションの予測**（本人 × 特定チームの相性）— これは"個人の特性"ではなく"関係性・カルチャー適合"であり、会社ごとにカルチャーが異なるため汎用実装は筋が悪い。個人スコア単独での予測はレッテル貼り（差別・法的リスク）に直結するため**やらない**。

代わりに、**「A の結果から、この人はどんなカルチャーの会社で活きるか」を一方向に導出**する。個人特性 → カルチャー親和性、という向きなら破綻しない。

## 命名と家族内の棲み分け

診断ファミリーは「働き方」系の名前が既に埋まっている：

| 診断 | 担当領域 | 問い |
|---|---|---|
| **playstyle（プレイスタイル／気質）** | 自分自身の働き方モード（個人の気質） | 「あなたは"どう"働くのが自然か」 |
| **thinking-style（思考スタイル）** | 認知（どう考える・学ぶか） | 「あなたは"どう"考えるか」 |
| **worklife-disposition（働き方の志向）** | 価値観・貢献の向き（5志向） | 「あなたは"何に"価値を感じ貢献するか」 |
| **teamwork-style（本 spec）** | **他者との"間"で起きる関わり方（対人の型）** | 「他者と一緒のとき"どう"関わるか」 |

- 呼称: **チームワーク・スタイル診断**（slug: `teamwork-style` / `survey_kind='teamwork_style'`）
- 「社会性」を避けた理由: "能力の高低"を連想させ評価的になるため。今回は非評価のスタイル論に寄せる。
- 「ワークスタイル」を避けた理由: playstyle（"どう働くか"）・worklife-disposition（"働き方の志向"）と意味が衝突するため。
- 棲み分けの原則: playstyle＝**自分自身の働き方モード**、teamwork-style＝**他者との間の関わり方**。両者の軸は交差しない（後述の軸設計で担保）。

## 診断モデル（3レイヤー）

### レイヤー1｜チームワーク・スタイル軸（非評価・4軸 → 16タイプ）

thinking-style（4軸×2極＝16型）と同じ骨格。すべて「良し悪しのない両極」。**playstyle の4軸（探索⇔深化／個人⇔協調／計画⇔即興／堅実⇔挑戦）とは一切交差しない、純・対人軸**に設計する。

| 軸 | 第1極 | 第2極 | 意味 |
|---|---|---|---|
| **率直さ** | 直言 | 調停 | 対立・指摘を正面から出すか、場を和らげ調整するか |
| **判断の重心** | 課題ドリブン | 関係ドリブン | 判断時に成果・正しさ優先か、人間関係・感情への配慮優先か |
| **距離感** | ドライ（公私分離） | ウェット（情緒的つながり） | 働く関係を仕事本位で築くか、情緒的な結びつきで築くか |
| **異論への構え** | 統一志向（揃える） | 多様許容（割れてよい） | チームで足並みを揃えたいか、意見が割れることを歓迎するか |

→ 2⁴ = 16 タイプ。各タイプに名前と「活きるカルチャー」を付与。

> 設計注記: ブレスト初期案にあった「協調↔自律」「構造↔即興」は playstyle の「個人⇔協調」「計画⇔即興」と被るため**除外**し、上記4軸に組み直した。

### レイヤー2｜成長ディメンション（発達的・"伸びしろ"・3項目）

EQ / SQ の中核。「高い方が一般に望ましい」＝ハレーションの芽が出る場所。スコアの高低で断じず、**本人向けの成長アドバイス**として提示する。

1. **自己認識** — 自分の感情・傾向・他者への影響をどれだけ客観視できているか
2. **他者視点の取得** — 相手の立場・感情を推し量る度合い
3. **感情の自己制御** — 対立・ストレス下で反応を制御できるか

> ハレーションの芽 ＝「特定タイプ × 低い自己認識 × カルチャー不一致」の重なりで顕在化する、という考え方。個人スコア単独では断定しない。

### レイヤー3｜カルチャー親和性（レイヤー1から導出）

タイプ軸を組織カルチャーの型へ写像（ライブ算出）。例：

- 直言 × 多様許容 → **議論歓迎・high-candor 型**（フラット）
- 調停 × 統一志向 → **合意形成・和型**（コンセンサス重視）
- ドライ × 課題 → **成果主義・プロフェッショナル型**
- ウェット × 関係 → **家族的・関係重視型**

## 設問形式（ハイブリッド）

社会性は最も"盛られやすい"領域のため、形式で妥当性を担保する。

- **レイヤー1（タイプ判定）= 二者択一（イプサティブ）**
  - 「どちらが自分に近い？」両方とも良く聞こえる2択。「全部当てはまる」を潰す。
  - 既存 `single_choice`（選択肢2つ）で表現可能。各選択肢を app 内スコアリングで軸の極へ写像（thinking-style の `answers.ts` と同型）。
- **レイヤー2（成長ディメンション）= 状況判断（SJT）数問**
  - 職場シーンを提示し「あなたならどうする」を選ぶ。自己申告が最も当てにならない自己認識等を、行動から推定する。
  - 同じく `single_choice`（設問 body にシーン記述、選択肢が対処行動）。app 内で発達スコアへ写像。
  - この SJT シナリオ・4軸語彙は**派生 spec B（面接官向け問診パターン）へそのまま流用**する。

いずれも**既存4階層スキーマ（survey → category → question → choice）のまま**表現でき、スキーマ拡張は `survey_kind` への値追加のみ。

## アーキテクチャ

thinking-style が確立した骨格を**加算的に複製**（既存機能は無改修）。

- **スキーマ**: `survey_kind` enum に `teamwork_style` を1値追加する migration のみ。テーブル追加・変更なし。
- **設問形式**: 二者択一・SJT ともに既存 `single_choice` で表現（スキーマ拡張不要）。
- **スコアリング/カルチャー親和性/成長アドバイス**: すべて app ローカルの純関数（ライブ算出、DB 永続化なし）。
  - `apps/candidate/app/_lib/teamwork-style/`（axes / score / answers / archetypes / culture-affinity / growth）
  - 型は app ローカル（クロスパッケージ消費者なし。`@bulr/types` に足さない）。
- **結果画面**: 新規ルート `/teamwork-style-diagnosis`（4軸バー ＋ アーキタイプカード ＋ カルチャー親和性 ＋ 成長アドバイス ＋ 共有パネル、none/partial/full 分岐）。
- **DB query**: `getTeamworkStyleSurveyId` / `getCandidateTeamworkStyleResponse`（thinking-style と同型）。
- **seed**: `kind='teamwork_style'` の survey を加算 seed。
- **導線**: 回答へ直行する deep-link ＋ ナビ入口。一覧除外フィルタ（`answered-surveys-query.ts`）へ `teamwork_style` を適切に扱わせる。

### 依存方向

types → db → ai → apps を厳守。本体は db query と app に閉じ、`@bulr/types` / `@bulr/ai` は変更しない。

## スコープ分割（2 spec）

| | spec | 置き場所 | 依存 |
|---|---|---|---|
| **A（本体・今回）** | `teamwork-style-diagnosis` | candidate app（診断ファミリー3つ目の原子診断） | thinking-style 骨格を加算複製 |
| **B（派生・次回）** | 面接官向け 社会性 問診パターン | business / assessment-engine | **A の4軸語彙と SJT シナリオを再利用** |

B は「A で作った SJT シーンと4軸の言葉」を面接官が対面で引き出す問診パターンへ焼き直す形。**A 完成後に着手**する。

## Goals

- チームワーク・スタイルを 4軸×2極＝16タイプ で決定論導出し、独立した結果体験（アーキタイプ・4軸可視化・カルチャー親和性・成長アドバイス・共有）を提供する。
- レイヤー2（成長ディメンション）を SJT で行動から推定し、自己申告の"盛り"に強くする。
- 既存の playstyle・thinking-style・worklife-disposition・スキルアンケート基盤の動作を変えない（加算のみ）。

## Non-Goals

- 診断結果の DB 永続化・履歴・版間比較（別 spec、self-analysis-history の系譜）。
- 会社ごとのカルチャー適合（B寄り）の汎用実装。個人 → カルチャー親和性の一方向導出のみ。
- 社会性スコアの高低による合否判定・レッテル付け（評価的スコアは作らない）。
- 面接官向け問診パターン（spec B）本体の実装。
- RPGクラス診断／診断アーキタイプへのチームワーク・スタイルの給餌（合成診断側の拡張）。

## 倫理・妥当性メモ

- **非評価トーンの徹底**: レイヤー1は両極とも価値中立。レイヤー2は"伸びしろ"＝本人向け成長文脈でのみ提示し、合否シグナルに転用しない。
- **社会的望ましさバイアス対策**: 二者択一（両選択肢とも好ましく聞こえる）＋ SJT（行動推定）で軽減。
- **予測の限界の明示**: 「特定タイプ × 低自己認識 × カルチャー不一致」の重なりで摩擦が顕在化しうる、という説明に留め、個人スコアでの断定はしない。

## Open Questions（spec 化フェーズで詰める）

- 各軸の設問数（二者択一の設問数）と、SJT の設問数（レイヤー2の推定に必要な最小数）。
- 16タイプそれぞれの名称・説明・カルチャー親和性コピー（コンテンツ作成コスト）。
- SJT シナリオの初期セット（spec B への流用を見据えた汎用性のあるシーン選定）。
- ナビ／一覧での並び順・導線（thinking-style と同様の deep-link 挙動の踏襲確認）。
