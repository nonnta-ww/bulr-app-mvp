# 評価ヒートマップ再設計 — Design Spec

**作成日**: 2026-05-18
**対象画面**: `/interviews/[sessionId]/report` の評価ヒートマップ部分
**対象コンポーネント**: `apps/web/app/(interviewer)/interviews/_components/heatmap.tsx`

---

## 1. 背景と問題

現状のヒートマップ（`heatmap.tsx`）は、6 カテゴリ × 5 次元 = **30 本の横棒バー** に加えて、スコープ分布・AI リテラシー分布・フリー質問数を平面的に並べた構造になっている。

面接官（主たる利用者）からのフィードバック：

> 今のヒートマップを見ても、どう判断したら良いのかわからない。

主な原因：

- **3 秒で全体像が掴めない**。30 本のバーがフラットに並んでいるため、どこを見ればよいのか優先順位がない。
- **比較基準がない**。スコア値（例: 真贋 1.5）が高いのか低いのかが分からない。
- **観察事実への接続がない**。「どのパターンで何が起きたか」がレポートから読み取れず、判断材料として弱い。
- **アクションに繋がらない**。「次に何を確認すべきか」「HR にどう共有するか」の判断につながらない。

---

## 2. 設計の指針

1. **判断は面接官に任せる**。LLM や UI は採用推奨・「強み / 弱み」のような評価ラベルを出さない（`product.md` 「AI は黒子、人間が決める」哲学に従う）。出すのは観察事実・スコア・到達状態のみ。
2. **3 秒 / 30 秒 / 3 分の階層**を持つ。3 秒で全体像、30 秒で根拠、3 分で個別ターン詳細へ。
3. **比較基準を内蔵**。Stage 1 はデータ蓄積前なので、ルーブリック定義の「標準的=2.0」「中堅射程=3.0」を固定ベンチマークとして表示。
4. **既存 AI サマリーテキストと役割分担**。ヒートマップは構造化指標、AI サマリーは自然言語ナラティブ。重複を避ける。

---

## 3. レイアウト全体像

│ │ │ 真贋 ━━●─ 2.3   │  │ 詰まり 2件   │      │ │
│ │ │ 判断 ━━●─ 2.0   │  │ 経験なし 1件 │      │ │
│ │ │ 射程 ━●━ 3.2   │  │ 未深掘り 1件 │      │ │
│ │ │ メタ ━●━━ 1.0   │  │ フリー 2件   │      │ │
│ │ │ AI  ━━●━ 2.5   │  └──────────────┘      │ │
│ │ └────────────────┘                          │ │
│ └────────────────────────────────────────────┘ │

│ │ [観察] [カバレッジ]                          │ │
│ └────────────────────────────────────────────┘ │
│ ┌─[アクティブタブの内容]────────────────────┐ │
│ │  観察タブ: 深掘り到達 / 詰まり・未到達 2列   │ │
│ │  または                                       │ │
│ │  カバレッジタブ: 6カテゴリ × パターングリッド │ │
│ └────────────────────────────────────────────┘ │
│ ┌─[AIサマリー（既存・このまま）]──────────────┐ │
│ │ 候補者は中堅エンジニアレベルの... (markdown) │ │

- **スティッキー判定サマリー** はビューポート上部に固定（`position: sticky; top: 0;`）。タブ切替時もスクロール時も常に見える。
- **タブ**は 2 つのみ:「観察」「カバレッジ」。デフォルトは「観察」。
- **AI サマリー**は既存 markdown のまま、ヒートマップの下に通常表示。
- **既存の「補足情報」セクション**（フリー質問数のみ表示）は削除。フリー質問数はスティッキー右側に統合済み。

## 4. スティッキー判定サマリー（A）

### 表示内容

| 要素 | 内容 |
|------|------|
| ヘッドライン | 「N パターン到達 / M パターン詰まり・未到達」（数値のみ、評価ラベルなし） |
| 5 次元バー | 真贋・判断力・射程・メタ認知・AI 活用、各バーに現スコア + ベンチマーク縦線 |
| 警告サイド | 「詰まり N 件 / 経験なし N 件 / 未深掘り N 件 / フリー質問 N 件」 |

### 色のルール

5 次元バーの **塗り色** はスコアの絶対値で決定:

- 0–3 軸（真贋・判断力・メタ認知・AI 活用）:
  - `>= 2.5`: 緑 (`#10b981`)
  - `>= 1.5`: 黄 (`#fbbf24`)
  - `< 1.5`: 赤 (`#ef4444`)
- 射程（1–5 軸）:
  - `>= 3.5`: 緑
  - `>= 2.5`: 黄
  - `< 2.5`: 赤

警告サイドの「詰まり N 件」は **N > 0 で赤字** にする。それ以外は通常色。

### ベンチマーク値（固定）

Stage 1 ではルーブリック定義に基づく固定値:

| 次元 | ベンチマーク | 根拠 |
|------|---|------|
| 真贋 | 2.0 | ルーブリック「標準的」 |
| 判断力 | 2.0 | 同上 |
| 射程 | 3.0 | プロダクト射程（中堅想定） |
| メタ認知 | 2.0 | 標準的 |
| AI 活用 | 1.5 | 「使ったことはあるが境界が曖昧」レベル |

将来（Stage 2 以降）データ蓄積後にパーセンタイル等への切替を検討するが、本スコープでは固定値で行く。値は `lib/heatmap-benchmark.ts` のような専用モジュールに切り出す。

### スティッキー高さ

スティッキー領域は概ね **140〜180px** に収める。ベンチマーク線の凡例（小さく「縦線 = ベンチマーク / 0–3 軸: 2.0 / 射程: 3.0」）を右下に表示。

## 5. 観察タブ（B-1: 深掘り到達 / 詰まり・未到達 2 列）

### 構造

```
┌─[深掘り到達 N件]──┐ ┌─[詰まり・未到達 M件]┐
│ [D-03] MS分割     │ │ [O-04] 採用判断     │
│  ●●●●●           │ │   選択肢が単一        │
│ [T-07] 障害切り分け│ │ [S-02] 秘密情報管理 │
│  ●●●○●           │ │   経験なし           │
│ ...               │ │ ...                  │
└──────────────────┘ └─────────────────────┘
```

### 振り分けルール

- **深掘り到達**: `pattern_coverage.stuck_type IS NULL` AND `level_reached >= 2` のパターン。
- **詰まり・未到達**: `stuck_type IS NOT NULL` または `level_reached <= 1` のパターン。

並び順は **両列とも `level_reached DESC, authenticity DESC`**（深掘り到達は深い順、詰まりは惜しい順）。

### 行の構成

**深掘り到達側**:
- パターンコード（`D-03` 等のチップ）
- パターン名（`pattern.title`）
- 5 次元ミニドット（5 個の小さな円、色はスコア値で緑/黄/赤、射程は別色帯）

**詰まり・未到達側**:
- パターンコード（グレーチップ）
- パターン名
- 詰まり種別（日本語ラベル化、後述）

### 詰まり種別ラベル

`stuck_type` enum → 日本語表示:

| enum 値 | 日本語ラベル |
|---|---|
| `not_experienced` | 経験なし |
| `shallow` | 浅い |
| `single_option` | 選択肢が単一 |
| `rigid` | 固執 |

ラベル変換は `lib/stuck-type-label.ts` のような専用ヘルパーで行う。

### クリック挙動

行クリックで **右からスライドインするドリルダウンパネル**（§7）を開く。

---

## 6. カバレッジタブ（C: パターン到達グリッド）

### 構造

カテゴリごとに行を持つ。各行は **カテゴリ名 + 進捗バー + 到達数バッジ** のヘッダーと、**そのカテゴリの全パターン**（実データではカテゴリにより 6〜15 個）をセルで並べたグリッドからなる。**グリッドは 12 列固定**で、12 を超えるカテゴリ（design=15, trouble=12 等）は 2 段目に折り返す。

```
システム設計       ▓▓▓▓▓░░░░░░░ 5/15 到達
              [D1][D2][D3][D4][D5][D6][D7][D8][D9][D10][D11][D12]
              [D13][D14][D15]

トラブル対応       ▓▓▓░░░░░░░░░ 3/12 到達
              [T1][T2][T3][T4][T5][T6][T7][T8][T9][T10][T11][T12]

...
```

### セルの色（到達度）

| 状態 | 色 |
|---|---|
| 未到達（カバレッジレコードなし） | 薄グレー `#f1f5f9` |
| 詰まり（`stuck_type` あり） | 中グレー `#94a3b8` |
| L1（`level_reached = 1`） | 薄赤 `#fca5a5` |
| L2 | 薄黄 `#fde68a` |
| L3 | 薄緑 `#6ee7b7` |
| L4（完走） | 緑 `#10b981` |

セル内には **パターンコードの末尾** （`D1`, `D2` 等）を小さく表示。

### 凡例

タブ末尾に色凡例を **常に表示**: 未到達 / 詰まり / L1 / L2 / L3 / L4。

### セルクリック

到達済みパターン（`stuck` 含む）のセルをクリック → §7 ドリルダウンパネル。未到達セルはクリック不可（カーソル `default`）。

---

## 7. ドリルダウンパネル（共通）

観察タブ・カバレッジタブの両方から、パターン行 / セルクリックで開く **右からスライドインするサイドパネル**。

### 表示内容

```
┌─[D-03 マイクロサービス分割]──[✕]┐
│                                  │
│ 到達段階  L4（メタ認知まで）       │
│ 真贋     3                       │
│ 判断力   3                       │
│ 射程     4                       │
│ メタ認知  3                       │
│ AI活用    3                       │
│                                  │
│ ─関連ターン (3件)─                │
│ Q1: 直近で分割判断したサービス…   │
│ Q2: 分割した粒度の選択肢は？       │
│ Q3: 100名規模なら判断は変わる？    │
│                                  │
│ ─評価メモ─                       │
│ 2023年のEC基盤分割について、QPS… │
└──────────────────────────────────┘
```

- 幅: 320–360px 程度
- 閉じる操作: `✕` ボタン / `Esc` キー / パネル外クリック
- スクロール: パネル内独立スクロール
- 詰まりパターンの場合、スコア部分は「—」表示、詰まり種別を強調表示

### データソース

- `pattern_coverage` レコードからスコア・notes
- `interview_turn` から関連ターン一覧（`pattern_id` で絞り込み）
- `assessment_pattern` （seed） からタイトル

## 8. データ層の拡張

現状の `HeatmapData`（`packages/types/src/evaluation.ts`）はカテゴリ集計のみ:

```ts
export interface HeatmapData {
  by_category: Record<PatternCategory, { avg_..., pattern_count }>;
  scope_distribution: ...;
  ai_literacy_distribution: ...;
  free_question_count: number;
}
```

新デザインで必要な追加データ:

1. **per-pattern 配列**: 全パターン（カバレッジ無も含む）の到達状態
2. **詰まり・経験なし・未深掘り のカウント**: スティッキー警告サイド用
3. **5 次元 session-wide 平均**: スティッキーバー用（by_category から重み付け平均で算出可だが、API 側で算出した方が一貫性が高い）

### 拡張案

```ts
export interface HeatmapData {
  // 既存
  by_category: Record<PatternCategory, CategoryAggregate>;
  scope_distribution: Record<1|2|3|4|5, number>;
  ai_literacy_distribution: Record<0|1|2|3, number>;
  free_question_count: number;

  // 追加
  overall: {
    avg_authenticity: number;
    avg_judgment: number;
    avg_scope: number;
    avg_meta_cognition: number;
    avg_ai_literacy: number;
    // 4つは互いに排他で合算するとセッション内の全カバレッジ件数になる
    reached_count: number;         // stuck_type IS NULL かつ level_reached >= 2  → 観察タブ「深掘り到達」
    stuck_count: number;           // stuck_type IN ('shallow','single_option','rigid')  → スティッキー「詰まり N件」
    not_experienced_count: number; // stuck_type = 'not_experienced'                      → スティッキー「経験なし N件」
    undeveloped_count: number;     // stuck_type IS NULL かつ level_reached <= 1          → スティッキー「未深掘り N件」
  };

  patterns: Array<{
    pattern_id: string;
    pattern_code: string;          // 'D-03' 等
    pattern_title: string;
    category: PatternCategory;
    level_reached: 0 | 1 | 2 | 3 | 4;
    stuck_type: StuckType | null;
    scores: {
      authenticity: number;
      judgment: number;
      scope: number;
      meta_cognition: number;
      ai_literacy: number;
    };
    notes: string;
    turn_count: number;
  }>;
}
```

注: `pattern_coverage.llm_evaluation` は NOT NULL かつ `SAFE_LLM_EVALUATION_FALLBACK` が全 0 を保証するため、`scores.*` と `notes` は非 null で十分。`not_experienced` パターンも `authenticity=0` 等の 0 値で記録されるため、null は構造的に発生しない。

`generateSessionReport` 関数（`packages/ai/src/functions/generate-session-report.ts`）の出力スキーマ（Zod）も同様に拡張する。

ドリルダウンパネルで関連ターンを表示するには、現状の `HeatmapData` 単体では足りないため:

- **方針 A（推奨）**: レポートページ側で `interview_turn` を別途取得（クライアント遷移時 or サーバ側プリフェッチ）。`HeatmapData` には turn_count のみ持つ。
- **方針 B**: `HeatmapData` に各パターンの関連ターン要約も含める（データ量増、再生成コスト増）。

**方針 A** を採用。ドリルダウン用に新 Server Action / RSC コンポーネント `<PatternDetail patternId={...} sessionId={...} />` を別実装。

### マイグレーション

既存セッションのレポート JSONB は **新スキーマと互換しないため**、以下のいずれか:

- **再生成（推奨）**: 既存セッションは旧 UI でフォールバック表示、新セッションのみ新 UI。期間限定で両方サポート。
- **バックフィル**: 既存 `pattern_coverage` テーブルから新 `HeatmapData` を再計算するスクリプトを書く。

→ 本スコープではバックフィルを実装する（Stage 1 のセッション数は限定的なので現実的）。スクリプトは `scripts/migrate-heatmap-v2.ts`。

## 9. AI サマリー・既存セクションの扱い

- **AI サマリーテキスト**: 既存のまま残す。ヒートマップ下に markdown 表示。
- **「補足情報」セクション**（現在フリー質問数のみ表示）: 削除。フリー質問数はスティッキー警告サイドに統合済み。
- **旧 30 本バーの per-category 詳細**: 削除。新カバレッジタブが「カテゴリごとのパターン到達状況」を可視化することで代替。次元レベルの per-category 詳細が必要な場合は admin パネルで確認する運用とする。

---

## 10. 採用判断・評価ラベルポリシー（再確認）

- **UI 上で「強み」「弱み」「中堅水準＋」のような評価ラベルを表示しない**。
- 表示するのは:
  - 数値スコア（真贋 2.3 等）
  - 到達状態（深掘り到達 / 詰まり・未到達）
  - 詰まり種別の事実ラベル（経験なし / 浅い / 選択肢が単一 / 固執）
  - ベンチマークとの位置関係（バーとベンチマーク線の視覚比較のみ）
- 色（緑/黄/赤）はスコアの絶対値で決まるアフォーダンスで、評価ラベルではない。
- 採用推奨・不採用推奨・保留などの判断ワードは UI・LLM 出力ともに **出さない**。

## 11. アクセシビリティ・印刷・モバイル

### モバイル

- 面接官は **PC 利用前提**（`product.md` の Stage 1 スコープ）。モバイル対応は **best-effort**。
- スティッキー判定: モバイルでも上部固定、ただし高さ抑制で 5 次元バーは横スクロール or 折り返し。
- タブは縦並びにフォールバックせず、横スクロール可能なタブバーに。
- カバレッジタブのグリッドは横スクロール可能に。

### 印刷 / PDF 出力

- 印刷時 CSS:
  - スティッキー解除（`position: static`）
  - 両タブの内容を **両方とも縦展開** （観察 → カバレッジの順）
  - ドリルダウンパネルは非表示
  - 色は最低限保持（ただし背景色は印刷で抜け落ちる前提で、コントラスト境界に依存しない設計）

### アクセシビリティ

- 色のみに依存しないよう、各バー・セルに必ず数値またはラベルテキストを併記。
- タブはキーボード操作可能（`role="tab"` / `aria-selected`）。
- ドリルダウンパネルは `role="dialog"` / `aria-modal="false"`（モーダルではないため）、初期フォーカスはタイトル、`Esc` で閉じる。

## 12. コンポーネント分割

```
apps/web/app/(interviewer)/interviews/_components/
├── heatmap.tsx                    # 旧、最終的に削除
└── report/
    ├── report-view.tsx             # ページ統合 (Client Component, タブ状態を持つ)
    ├── verdict-summary.tsx         # スティッキー A
    ├── observation-tab.tsx         # B-1 タブ本体
    ├── coverage-tab.tsx            # C タブ本体
    ├── coverage-cell.tsx           # カバレッジ単一セル
    ├── pattern-row.tsx             # 観察タブの行
    ├── pattern-detail-panel.tsx    # ドリルダウンサイドパネル (RSC or async client)
    └── benchmarks.ts                # ベンチマーク定数
```

タブ切替は `report-view.tsx` の `useState` で十分（URL 同期は **しない**、リロードでデフォルトの「観察」に戻る）。

`pattern-detail-panel.tsx` は React Server Component として実装し、クリック時に Server Action でデータ取得 → クライアント側でパネル開閉アニメーション、という構成を検討。複雑なら一旦全クライアントで OK。

## 13. スコープ外（将来検討）

- **パーセンタイル / 候補者間比較ベンチマーク**: Stage 2 以降、データ蓄積後。
- **モバイル最適化**: best-effort のみ、専用 UI なし。
- **タブ URL 同期 / 共有可能 URL**: しない（リロードで初期タブに戻る）。
- **ドリルダウン内での編集**（手動評価入力）: admin パネルの責務、本画面はリードオンリー。
- **カバレッジタブからのフィルタ / ソート**: 行追加機能。本スコープでは表示のみ。
- **過去セッションとの推移グラフ**: Stage 2。
- **エクスポート（CSV / PDF）ボタン**: 現状の admin 機能で代替、本画面には追加しない。

---

## 14. オープン論点（実装中に判断する細部）

- スティッキー判定とタブバーの間のシャドウ / ボーダーの強さ（スクロール感の演出）。
- ベンチマーク線の凡例位置（スティッキー内 vs ツールチップ）。
- 詰まり種別のソート順（経験なし → 浅い → 選択肢が単一 → 固執 の順か、データ準拠か）。

## 15. 受け入れ条件 (Acceptance Criteria)

実装完了の判定基準:

1. レポートページにアクセスすると、スティッキー判定サマリー + タブ + AI サマリーの新レイアウトが表示される。
2. スティッキー判定が常に上部に固定され、スクロールしても見える。
3. 5 次元バーがスコア絶対値に応じた色（緑/黄/赤）で表示され、ベンチマーク縦線が表示される。
4. 警告サイドに詰まり・経験なし・未深掘り・フリー質問の件数が出る。詰まり > 0 のとき赤字。
5. タブ切替（観察 ↔ カバレッジ）が動作する。
6. 観察タブで深掘り到達 / 詰まり・未到達の 2 列が、振り分けルールに従って表示される。
7. カバレッジタブで 6 カテゴリ × パターンのグリッドが表示され、色凡例が出る。
8. 任意のパターン行 / セルをクリックすると、ドリルダウンパネルが右からスライドインする。Esc で閉じる。
9. AI サマリーが既存通り下部に表示される。
10. 旧「補足情報」セクションが削除されている。
11. 既存セッションのレポートも、マイグレーションスクリプト実行後に新 UI で表示される。
12. PC ブラウザ（Chrome / Safari 最新）で表示崩れがない。
13. 評価ラベル（「強み」「弱み」「中堅水準＋」等）が UI 上に出ていない。
14. 印刷プレビューで両タブの内容が縦展開され、ドリルダウンパネルが非表示になる。
