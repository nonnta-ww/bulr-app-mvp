# Requirements Document

## Introduction

本機能は、面接録音キャプチャの前提となる「候補者の同意」を実効化する。現状は全セッションが作成と同時に自動的に同意済みと見なされ、同意ゲートが構造的に素通り（vacuous）している。これを、面接官が同意文を確認し「候補者から録音同意を口頭で得た」と明示的に確定する**面接官アテステーション**方式に置き換え、同意記録が無ければキャプチャを開始しないようにする。同意モデルはハイブリッド（案C）で、今回は面接官アテステーションのみ実装し、記録は将来の候補者セルフ同意を追加できる形で保持する。本機能は `realtime-interview-capture` の Req 1.6 / Req 7.5 を実効化するものであり、Wave 7 の本番投入（実候補者の面接録画）を法務/プライバシー上「回せる」状態にすることを目的とする。

## Boundary Context

- **In scope**: 面接官アテステーションによる同意取得と記録、同意ゲートの実効化（未同意ならキャプチャ開始不可）、版管理された同意文の提示、既存/新規セッションを未同意状態で扱うこと、同意記録操作の権限と冪等性。
- **Out of scope**: 候補者セルフ同意フローの実装、同意の撤回/取り消しフロー、削除請求対応、同意文言の法務確定・レビュー、ja-v1 以外の多言語版同意文の起稿。
- **Adjacent expectations**:
  - `realtime-interview-capture` の既存キャプチャ開始ゲートを実効化する（新たな別ゲートは設けない）。
  - `session-from-entry` / エントリー作成フローは、面接セッションを未同意状態で開始させることを期待する。
  - 同意文に記載する保持期間は、音声データの自動削除ポリシー（30 日）と整合させる。
  - 将来の候補者セルフ同意は、記録された「取得方法」を拡張して載せられることを想定する（本 spec では動線を実装しない）。

## Requirements

### Requirement 1: 同意ゲート（キャプチャ開始の前提）

**Objective:** As a 運営者, I want 同意取得の記録が無いセッションで録音キャプチャが始まらないこと, so that 候補者の同意なき録画を構造的に防げる

#### Acceptance Criteria

1. While セッションに同意取得の記録が無い, the system shall キャプチャの開始を許可しない
2. If 面接官が同意取得の記録が無いセッションでキャプチャ開始を試みた, then the system shall 開始を拒否し、同意が必要である旨を面接官に表示する
3. When セッションに同意取得の記録がある状態でキャプチャ開始が指示された, the system shall キャプチャの開始を許可する
4. The system shall オンライン（会議ボット）と対面（端末録音）のいずれのキャプチャ経路にも同一の同意ゲートを適用する

### Requirement 2: 面接官による同意取得（アテステーション）

**Objective:** As a 面接官, I want キャプチャ開始前に同意文を確認し「候補者から録音同意を口頭で得た」と明示的に確定できること, so that 実面接で正当な同意記録を残せる

#### Acceptance Criteria

1. While セッションに同意取得の記録が無い, the system shall キャプチャ開始操作の前段として同意ステップ（同意文の提示と明示的な同意確認）を面接官に提示する
2. While 面接官が同意を確定していない, the system shall キャプチャ開始の操作を無効化する
3. When 面接官が同意文を確認し「候補者から録音同意を口頭で得た」ことを明示的に確定した, the system shall 当該セッションの同意取得を記録する
4. The system shall 同意の確定を、既定値や自動設定ではなく面接官の明示的な操作によってのみ発生させる

### Requirement 3: 同意記録の内容と provenance

**Objective:** As a 運営者, I want 各同意記録が「いつ・どの版で・どの方法で・誰が」取得したかを保持すること, so that 監査や法務対応時に同意の正当性を確認できる

#### Acceptance Criteria

1. When 同意が取得された, the system shall 同意取得の日時を記録する
2. When 同意が取得された, the system shall 同意した同意文の版を記録する
3. When 同意が取得された, the system shall 同意取得の方法を記録し、面接官アテステーションが他の取得方法と区別可能であるようにする
4. When 面接官アテステーションで同意が取得された, the system shall 同意を確定した面接官の識別情報を記録する

### Requirement 4: 同意文（版管理）の提示と内容

**Objective:** As a 面接官, I want 提示される同意文が版管理され、録音対象・利用目的・保持期間・データの扱いを明示すること, so that 何に対する同意かを明確にして候補者に説明できる

#### Acceptance Criteria

1. When 同意ステップが提示された, the system shall 現行版の同意文を面接官に表示する
2. The system shall 同意文に、録音対象・利用目的・データ保持期間・データの取り扱いを記載する
3. The system shall 同意文を版で識別し、記録される同意版が面接官へ提示された版と一致するようにする
4. The system shall 同意文に記載する保持期間を、音声データの自動削除ポリシー（30 日）と整合させる

### Requirement 5: 既存セッションと新規作成の整合

**Objective:** As a 運営者, I want 本機能導入前に自動的に同意済みとされていたセッションが以後は明示同意を要すること, so that 骨抜きだった同意が遡って実効化される

#### Acceptance Criteria

1. Where 本機能導入前に作成されたセッション, the system shall 同意取得の記録が無い状態として扱い、キャプチャ開始前に同意を要求する
2. When 新規に面接セッションが作成された, the system shall 同意取得の記録が無い状態で当該セッションを開始する

### Requirement 6: 同意記録操作の権限と冪等性

**Objective:** As a 運営者, I want 同意記録操作が正当なセッション担当者のみ実行でき、重複実行でも記録が壊れないこと, so that 同意記録の完全性が保たれる

#### Acceptance Criteria

1. If 同意記録の操作が当該セッションを担当していない利用者から行われた, then the system shall その操作を拒否する
2. When 既に同意が記録されているセッションで同意記録操作が再度行われた, the system shall 記録を破壊せず整合した結果を返す
3. The system shall 同意記録の各項目（日時・版・方法・アテスター）を一貫した単位で記録し、一部だけが欠けた記録を残さない
