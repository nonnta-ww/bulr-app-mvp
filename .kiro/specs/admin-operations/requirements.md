# 要件定義書

## プロジェクト説明

**対象ユーザー**: 運営者（admin / 創業者）

**現状**: `apps/admin`（admin.bulr.net）は `admin-review-panel` の成果（BtoB セッション一覧・詳細・手動評価・CSV/JSON エクスポート）が移設済みで、`requireAdmin` ガードも整備済み。しかし候補者・企業データの管理 UI、マスタ CMS、LLM コスト監視はすべて未実装。マスタはシードスクリプト経由でしか変更できない。

**変えること**: 運営者が `admin.bulr.net` にログインし、（1）候補者プロフィールの閲覧・無効化・クォータリセット、（2）企業の閲覧・無効化・新規作成、（3）`skill_survey` 設問・選択肢の CMS 編集、（4）`assessment_pattern` の閲覧（読み取り専用）、（5）LLM コストとクォータ使用状況の監視ダッシュボード——をすべて `apps/admin/app/` から利用できる。

---

## 要件

### 1. 候補者管理

**1.1** システムが `GET /candidates` にアクセスされたとき、`requireAdmin()` によって保護された候補者一覧ページを表示しなければならない。一覧には候補者 ID、表示名、メールアドレス、登録日時、有効/無効状態、当月の模擬面接クォータ使用数（`mock_interview` テーブルの当月件数）、スキルアンケート完了済みフラグを含まなければならない。

**1.2** システムが候補者一覧ページを表示するとき、氏名またはメールアドレスによるキーワード検索、`is_active` による有効/無効フィルタ、ページング（1 ページ 50 件）をサポートしなければならない。

**1.3** システムが `GET /candidates/[id]` にアクセスされたとき、候補者詳細ページを表示しなければならない。詳細には候補者プロフィール情報、紐づく履歴書ドキュメント一覧、スキルアンケート回答サマリー、模擬面接履歴（パターンコード・実施日・ターン数）を含まなければならない。

**1.4** システムが `disableCandidateProfile` アクションを実行するとき、`candidate_profile.is_active` を `false` に設定しなければならない。物理削除は行わない。

**1.5** システムが `resetCandidateQuota` アクションを実行するとき、対象候補者の当月分 `mock_interview` レコードを削除することなく、クォータカウントをリセット（月次クォータリセットマーカーを更新）できる。具体的実装は設計フェーズで決定する。ただし実際の `mock_interview` レコードは削除しない。

> **境界補足**: クォータ enforcement のロジック（「3 回/月を超えたらセッション作成を拒否する」判定）は `mock-interview` spec が所有し、本 spec はリセットと可視化のみを担う。

**1.6** システムが `disableCandidateProfile` または `resetCandidateQuota` を実行するとき、`requireAdmin()` による認証ガードを必須とし、実行結果（成功/失敗）を明示的に UI でフィードバックしなければならない。

---

### 2. 企業管理

**2.1** システムが `GET /companies` にアクセスされたとき、`requireAdmin()` によって保護された企業一覧ページを表示しなければならない。一覧には企業 ID、企業名、登録日時、有効/無効状態（`is_active`）、紐づく募集件数（`opening` テーブルの件数）を含まなければならない。

**2.2** システムが企業一覧ページを表示するとき、企業名によるキーワード検索、`is_active` フィルタ、ページング（1 ページ 50 件）をサポートしなければならない。

**2.3** システムが `GET /companies/[id]` にアクセスされたとき、企業詳細ページを表示しなければならない。詳細には企業名、登録情報、紐づく募集（`opening`）の一覧、所属する面接官（`user_profile` 経由で `company_id` が一致するユーザー）の一覧を含まなければならない。

**2.4** システムが `createCompany` アクションを実行するとき、企業名を入力として `company` テーブルに新規レコードを挿入し、作成した企業の詳細ページへリダイレクトしなければならない。

**2.5** システムが `disableCompany` アクションを実行するとき、`company.is_active` を `false` に設定しなければならない。物理削除は行わない。

**2.6** システムがすべての企業変更操作（作成・無効化）を実行するとき、`requireAdmin()` による認証ガードを必須とし、実行結果を UI でフィードバックしなければならない。

---

### 3. スキルアンケート CMS

**3.1** システムが `GET /masters/skill-survey` にアクセスされたとき、`requireAdmin()` によって保護されたスキルアンケートマスタ一覧ページを表示しなければならない。一覧には全サーベイ（`skill_survey` テーブル）、職種名、有効/無効状態を含まなければならない。

**3.2** システムが `GET /masters/skill-survey/[surveyId]` にアクセスされたとき、選択されたサーベイの完全ツリー構造（カテゴリ → 設問 → 選択肢）を表示しなければならない。

**3.3** システムが設問編集フォームを提出するとき、`updateSkillSurveyQuestion` アクションによって `skill_survey_question.body`、`question_type`、`display_order` を更新できなければならない。

**3.4** システムが選択肢編集フォームを提出するとき、`updateSkillSurveyChoice` アクションによって `skill_survey_choice.label`、`display_order` を更新できなければならない。

**3.5** システムがスキルアンケート CMS の編集操作を実行するとき、`requireAdmin()` による認証ガードを必須とし、実行結果を UI でフィードバックしなければならない。

**3.6** スキルアンケート CMS において、新規カテゴリの作成・既存カテゴリの削除・設問の削除・サーベイ自体の作成/削除は本 spec のスコープ外とする（読み取りと既存レコードの編集のみ）。

---

### 4. アセスメントパターン閲覧（読み取り専用）

**4.1** システムが `GET /masters/assessment-pattern` にアクセスされたとき、`requireAdmin()` によって保護されたアセスメントパターン一覧ページを表示しなければならない。一覧にはパターンコード、カテゴリ、タイトル、有効/無効状態を含まなければならない。

**4.2** システムが `GET /masters/assessment-pattern/[code]` にアクセスされたとき、選択されたパターンの詳細（コード、カテゴリ、タイトル、説明、4 段階の質問テンプレート、シグナル、AI 観点）を表示しなければならない。

**4.3** アセスメントパターンの CUD 操作（作成・更新・削除）は本 spec のスコープ外とする。閲覧のみを提供する。

---

### 5. LLM コスト・クォータ監視

**5.1** システムが `GET /monitoring` にアクセスされたとき、`requireAdmin()` によって保護された LLM コストダッシュボードを表示しなければならない。ダッシュボードには以下の集計値を含まなければならない。
- 全 `mock_interview` レコードの `metadata.llm_cost_estimate.estimated_usd` の合計（全期間）
- モデル別内訳（MVP では Claude Sonnet 4.6 のみ）
- 機能別内訳（mock-interview）
- 候補者別コスト上位 10 名
- 直近 30 日間の日次コストトレンド（日付と金額）

**5.2** システムが LLM コストデータを集計するとき、`mock_interview.metadata` JSONB フィールドの `llm_cost_estimate.estimated_usd`、`llm_cost_estimate.input_tokens`、`llm_cost_estimate.output_tokens` を参照しなければならない。Anthropic 課金 API は使用しない。

**5.3** システムが `GET /monitoring/quota` にアクセスされたとき、候補者別クォータ使用状況ページを表示しなければならない。表示内容には候補者名、メールアドレス、当月の模擬面接実施回数、月次上限（3 回）に対する残回数、最終実施日を含まなければならない。

**5.4** システムがクォータ使用状況を表示するとき、実施済み回数が上限（3 回）に達している候補者を視覚的に強調（例: バッジ表示）しなければならない。

**5.5** 監視ダッシュボードはすべて読み取り専用である。LLM コストの手動調整、Anthropic 課金 API との突合、予測分析・異常検知は本 spec のスコープ外とする。

---

### 6. 共通・横断要件

**6.1** システムが `apps/admin/app/` 配下のすべての新規ルートにアクセスされたとき、既存の `requireAdmin()` ガードによって認証を強制しなければならない。既存のセッション検証 UI（`/sessions`、`/sessions/[id]`）の動作を変更してはならない。

**6.2** システムが候補者または企業を無効化するとき、物理削除（ハードデリート）を行わず、`is_active` フラグを `false` に設定するソフトデリートを使用しなければならない。

**6.3** `candidate_profile` および `company` テーブルに `is_active` boolean カラムが存在しない場合、本 spec の実装前に Drizzle マイグレーションによって追加しなければならない。

**6.4** システムが新規の集約クエリを追加するとき、`packages/db/src/queries/admin/` ディレクトリに配置し、既存の `session-list-query.ts`・`session-detail-query.ts` と同じバレルチェーン（`admin/index.ts` → `queries/index.ts`）に乗せなければならない。

**6.5** システムが変更系操作（Server Action）を実行するとき、入力は Zod で検証し、`requireAdmin()` または `adminAction` ラッパーによって認可を確認しなければならない。

**6.6** 新規 UI コンポーネントは `apps/admin/app/_components/` 配下に配置し、ページ固有のアクションは `apps/admin/app/(該当ルート)/_actions/` 配下に配置しなければならない。
