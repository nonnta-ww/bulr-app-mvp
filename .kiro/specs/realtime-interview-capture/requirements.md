# Requirements Document

## Project Description (Input)

面接中の面接官操作をゼロにするリアルタイム面接キャプチャ機能。現行 assessment-engine のターン単位手動録音（状態A/B ステートマシン + ターン毎 Whisper バッチ処理）を置き換える。

構成要素:

1. **ミーティングボット** — Zoom / Google Meet / Teams に bulr ボットが自動参加して音声を取得（Recall.ai 等のボットインフラ API を利用、buy not build。対面面接向けにブラウザマイク常時録音のフォールバックも提供）
2. **リアルタイム文字起こし＋話者分離** — Deepgram / AssemblyAI / Gladia 等の日本語対応ストリーミング STT で面接官/候補者を分離したライブトランスクリプトを生成
3. **サイドパネル UI** — 面接官は何も操作しない。bulr が会話を聞きながら、57 状況パターンのカバレッジ進捗と「次の深掘り質問候補 3 つ」をサイドパネルに継続表示し、面接官は見たいときだけ見る。現在の状態A（録音中）/状態B（候補選択）UI は廃止

既存資産の assessment_pattern / pattern_coverage / 5次元スコアリング / session_report / LLM 評価パイプラインは維持し、入力ソースをターン録音からライブトランスクリプトのセグメントに差し替える。面接終了後は既存の面接後レポート生成につなげる。対象アプリは apps/business。Stage 1 検証（実面接での利用）を可能にすることがゴール。

## Introduction

本機能は、bulr の面接アシスタントにおける「面接中の面接官操作」をゼロにするキャプチャ層の刷新である。現行のターン単位手動録音（状態A/B）では、面接官が録音開始・質問送り・候補選択の操作を面接中に強いられ、候補者との対話への集中を阻害している。本機能では、オンライン面接では録音ボットが会議に自動参加し、対面面接では端末マイクの連続録音により、会話が話者分離つきでリアルタイムに文字起こしされる。面接官の画面には、操作を要求しないサイドパネルとしてパターンカバレッジ進捗と次の質問候補が会話に追従して自動更新される。既存の評価パイプライン（57 状況パターン × 4 段階深掘り、5 次元スコアリング、パターンカバレッジ集約、面接後レポート）は出力形式を変えずに維持し、入力のみをライブトランスクリプトに差し替える。

## Boundary Context

- **In scope**: 面接実施フロー（キャプチャ開始〜面接後レポート生成）の刷新。録音ボットの会議参加、対面向け連続録音、リアルタイム文字起こしと話者分離、操作不要のサイドパネル表示、ライブトランスクリプトの評価パイプライン接続、旧方式データの閲覧互換。対象は企業向けアプリの面接官 UI。
- **Out of scope**: ATS・カレンダー連携、レポートの共有 URL 発行・ハイライトクリップ作成、複数面接官の同席支援・面接官間評価ばらつき分析、候補者向け画面・候補者への結果開示、模擬面接（候補者向けアプリ）への適用、映像（録画）の取得・解析、面接スケジューリング、課金。
- **Adjacent expectations**: 57 状況パターンと 4 段階深掘りテンプレ（assessment-pattern-seed）は変更せず利用する。エントリーからのセッション作成（session-from-entry）の候補者情報引き継ぎとパターン選定支援を前提とする。管理画面（admin-review-panel / admin-operations）は新方式セッションのデータを既存機能で閲覧・評価できることを期待する。音声の 30 日後自動削除の既存運用は本機能のデータにも適用される。

## Requirements

### Requirement 1: 面接キャプチャの開始（録音ボット / 対面録音）

**Objective:** As a 面接官, I want 会議 URL の指定または対面録音の開始だけで面接キャプチャが始まること, so that 面接中にツール操作へ注意を割かず候補者との対話に集中できる

#### Acceptance Criteria

1. When 面接官がオンライン会議の URL を指定してキャプチャ開始を指示した, the system shall 当該会議に録音ボットを参加させ、参加成功を面接官の画面に表示する
2. The system shall Zoom / Google Meet / Microsoft Teams の会議 URL を受け付ける
3. When 録音ボットが会議に参加した, the system shall 記録中であることが会議参加者に識別できる名称でボットを表示させる
4. If 録音ボットが会議に参加できない（URL 不正・入室拒否・会議未開始等）, the system shall 失敗理由を面接官に表示し、再試行または対面録音への切替を選択できるようにする
5. Where 対面面接, the system shall 面接官の端末マイクによる連続録音でキャプチャを開始できるようにする
6. While セッションに同意取得の記録が無い, the system shall キャプチャを開始しない
7. When キャプチャが開始された, the system shall 面接セッションのステータスを進行中に更新する

### Requirement 2: リアルタイム文字起こしと話者分離

**Objective:** As a 面接官, I want 面接の会話が話者別にリアルタイムで文字起こしされること, so that 録音操作やメモ取りなしで会話が記録され、評価の根拠として参照できる

#### Acceptance Criteria

1. While キャプチャ中, the system shall 発話内容を逐次文字起こしし、発話からおおむね 10 秒以内にライブトランスクリプトとして面接官の画面に表示する
2. The system shall 文字起こしを面接官の発話と候補者の発話に分離してラベル付けする
3. If 話者の自動判別に十分な確信が得られない, the system shall 当該発話を話者未確定としてラベル付けし、評価処理では除外せずに扱う
4. The system shall 日本語の面接会話を文字起こしできる
5. If 文字起こし処理が一時的に中断した, the system shall 自動的に再開を試み、面接の進行を妨げずに復旧状況を面接官に表示する
6. If 文字起こしが復旧不能になった, the system shall 音声の録音を継続し、面接終了後に未処理区間を文字起こしして評価に反映する
7. The system shall キャプチャした音声を保存し、既存の保持ポリシー（30 日後自動削除）の対象とする

### Requirement 3: 操作不要のサイドパネル（カバレッジ進捗と質問候補）

**Objective:** As a 面接官, I want パターン消化状況と次に聞くべき質問候補が会話に追従して自動更新されること, so that 画面操作なしで深掘りの質を保てる

#### Acceptance Criteria

1. While 面接中, the system shall 計画パターンの消化状況（カバー済み / 進行中 / 未着手）と現在パターンの到達段階を自動更新して表示する
2. While 面接中, the system shall 次の質問候補を常時 3 件表示し、会話の進行に応じて自動更新する
3. When 候補者の回答の一区切りを検知した, the system shall おおむね 15 秒以内に質問候補とカバレッジ表示を更新する
4. The system shall 質問候補のうち少なくとも 1 件を未カバーパターンへの移行質問とする
5. The system shall キャプチャの開始と終了以外に面接官の操作を必須としない
6. The system shall 面接官が提示候補のいずれを使ったか、または独自質問をしたかを会話内容から自動判別する（手動選択を要求しない）
7. When 面接官が 57 パターンに該当しない独自質問をした, the system shall 当該質問・回答をフリー質問として記録し、パターン評価集約には含めず面接後レポートの総評に反映する
8. While 面接中, the system shall 経過時間と残り計画パターン数を表示する

### Requirement 4: ライブトランスクリプトの評価パイプライン接続

**Objective:** As a 採用責任者, I want 既存の 5 次元評価・パターンカバレッジ・レポートがライブトランスクリプトを入力として変わらず機能すること, so that 過去セッションとの比較可能性と評価の一貫性が保たれる

#### Acceptance Criteria

1. The system shall ライブトランスクリプトを質問と回答の単位に自動分割し、評価分析の入力とする
2. The system shall 各質問・回答単位を 57 状況パターンのいずれか、またはフリー質問に自動分類する
3. When 1 つのパターンに対する深掘りが完了したと判定された, the system shall 当該パターンのカバレッジ評価（5 次元スコア・到達段階・詰まり種別）を集約して更新する
4. The system shall 評価の出力形式（5 次元 0–3 スコア、到達段階 0–4、詰まり種別）を既存セッションと同一に保つ
5. If セッションあたりの自動解析回数の上限に達した, the system shall 以降の自動解析を停止して面接官に通知し、録音と文字起こしは継続する

### Requirement 5: 面接終了とレポート生成

**Objective:** As a 面接官, I want 面接終了の操作ひとつで既存の面接後レポートまで自動生成されること, so that 面接直後に評価結果を確認・共有できる

#### Acceptance Criteria

1. When 面接官が面接終了を指示した, the system shall キャプチャを停止し、録音ボットが会議参加中であれば退出させる
2. When 録音ボットが会議の終了を検知した, the system shall キャプチャを自動停止し、面接官に終了を通知する
3. When キャプチャが停止した, the system shall 未集約のパターンカバレッジを集約したうえで、既存形式の面接後レポート（ヒートマップ＋サマリーテキスト）を生成する
4. The system shall 面接終了後に全文トランスクリプトを話者ラベル付きでセッション詳細から閲覧できるようにする
5. If レポート生成が失敗した, the system shall キャプチャ済みデータを保持したまま再実行手段を面接官に提供する

### Requirement 6: 既存方式からの移行と互換性

**Objective:** As a 運営者, I want 旧ターン録音方式のデータが引き続き閲覧でき、新規面接は新方式に一本化されること, so that 検証データの連続性を保ちながら UI の複雑性を増やさない

#### Acceptance Criteria

1. When 新規の面接セッションを開始する, the system shall 新キャプチャ方式（録音ボット / 対面録音）のみを提供する（旧 状態A/B UI は提供しない）
2. The system shall 旧方式で実施済みのセッションのレポート・回答記録の閲覧を引き続き可能とする
3. The system shall エントリーからのセッション作成（候補者情報の引き継ぎとパターン選定支援）を新方式でも維持する
4. The system shall 管理画面の回答全文確認・手動評価入力・LLM 評価突合・エクスポートを新方式のセッションでも利用可能とする

### Requirement 7: セキュリティ・プライバシー運用

**Objective:** As a 運営者, I want 録音・文字起こし・解析が既存のセキュリティと同意の統制下で動作すること, so that 候補者の個人情報と面接データが保護される

#### Acceptance Criteria

1. The system shall 音声・トランスクリプト・評価データへのアクセスをセッション所有者である面接官と管理者に限定する
2. The system shall 会議 URL を含む全ての外部入力を検証し、不正な形式を拒否する
3. The system shall 録音ボット経由で取得した音声にも 30 日後自動削除を適用する
4. The system shall トランスクリプトと評価データを音声削除後も保持する
5. The system shall 同意記録（取得日時・同意文バージョン）をセッションに保持する
6. When 面接官がキャプチャの中止を指示した, the system shall 録音・文字起こし・自動解析を即時停止する
7. When 面接官がキャプチャの中止を指示した, the system shall セッションを中断扱い（status='abandoned'）に更新し、面接セッション一覧で「中断」と表示する（in_progress のまま残して実態とズレるのを防ぐ）

### Requirement 8: 信頼性

**Objective:** As a 面接官, I want 30〜40 分の実面接で安定して動作すること, so that 本番の採用面接に安心して導入できる

#### Acceptance Criteria

1. The system shall 60 分連続のキャプチャを処理できる
2. If 面接官の画面が一時的に切断・リロードされた, the system shall 録音ボット経由のキャプチャを中断せず、画面再表示時に最新の進捗・トランスクリプト・質問候補を復元して表示する
3. While 対面録音中, if 端末のネットワークが一時的に切断された, the system shall 切断中の音声を失わずに保持し、再接続後に未送信区間を処理する

### Requirement 9: 面接の一時停止と再開

**Objective:** As a 面接官, I want 面接を一時的に中断して後で同じセッションを続行できること, so that 休憩や録音すべきでない会話の発生時に、緊急停止（中止）せずに面接を続けられる

**背景:** 緊急停止用の「中止」（Req 7.6, 不可逆・再開不可）とは別物の「再開可能な一時停止」。録音ボットは通話に残したまま、一時停止中の発言は記録せず AI 解析を止める（A案: 解析停止＋停止中の発言は破棄）。

#### Acceptance Criteria

1. When 面接官が録音中（recording）に一時停止を指示した, the system shall capture_status を paused に遷移し、録音ボットは通話に残したまま AI 解析を停止する
2. While 一時停止中（paused）, the system shall 受信したトランスクリプト（ボット経由・対面マイク双方）を永続化せず破棄する
3. While 一時停止中（paused）, the system shall 自動解析（論理ターン確定・質問候補生成）を停止する
4. When 面接官が一時停止中に再開を指示した, the system shall capture_status を recording に戻し、同じセッションを続行する（経過時間・トランスクリプト・カバレッジは保持）
5. While 一時停止中（paused）, if 録音ボットの status イベントが届いた, the system shall それを破棄し、勝手に再開（recording 遷移）しない（再開はユーザー操作のみ）
6. When 面接官が一時停止中に面接終了または中止を指示した, the system shall それぞれ通常の終了（stopping）・中止（aborted）として処理する
