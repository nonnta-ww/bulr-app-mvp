# Evaluation Rubric

問診回答の評価基準を体系化する。LLM 自動評価（`evaluateAnswer` ツール）と創業者の手動評価の **両方** を `assessment_answer` レコードに保存し、Stage 1 ではその一致度を検証する。

詳細な深掘り質問テンプレと真贋シグナルは `docs/03-probe-logic.md` を参照。本ドキュメントは LLM プロンプトと DB スキーマの上位規約を固定する。

## 5 次元スコア

各パターンの回答に対し、以下 5 次元のスコアを付与する。

| 次元 | カラム名 | 範囲 | 何を測るか |
|---|---|---|---|
| 真贋 | `authenticity` | 0-3 | 経験が本物か（具体性・時系列・関係者・捨てた仮説・後悔） |
| 判断力 | `judgment` | 0-3 | トレードオフを認識しているか（選択肢の数・軸の言語化・定量定性） |
| 射程 | `scope` | 1-5 | 判断のスケール（1=タスク / 2=機能 / 3=プロダクト / 4=事業 / 5=組織） |
| メタ認知 | `meta_cognition` | 0-3 | 文脈変化への感度（規模・フェーズ・バイアス自覚） |
| AI 活用 | `ai_literacy` | 0-3 | AI を使う / 使わない境界の判断軸 |

スコア値は **整数のみ**。0 = 該当なし / 詰まり、1 = 限定的、2 = 標準的、3 = 高い。射程のみ 1-5 の段階値。

---

## 1. 真贋スコア（authenticity）

経験が本物かを 5 つの観察項目で判定する。

| 観察項目 | 経験者の典型 | 非経験者・伝聞・盛りの典型 |
|---|---|---|
| 具体性 | 「2022 年に〜のサービスで、QPS が 800 から 3000 に」 | 「ある程度の規模で」「結構な負荷で」 |
| 時系列 | 検知 → 仮説 → 検証 → 対処 が一貫 | 順序が後付けで矛盾、検証ステップ抜け |
| 関係者 | 「DBA の〇〇さんと相談、PdM が判断」 | 「チームで」「上の人と相談」 |
| 捨てた仮説 | 「最初は〜を疑ったが、〜の理由で違うと判断」 | 一直線に正解にたどり着く語り |
| 後悔 | 「今思えば〜すべきだった」 | 「ベストを尽くした」「問題なかった」 |

**スコアリング基準**:
- 3: 5 項目のうち 4 つ以上を満たす
- 2: 3 つ満たす
- 1: 1〜2 つ満たす
- 0: 全て満たさない、または明確に矛盾

---

## 2. 判断力スコア（judgment）

トレードオフ認識の深さを 4 つの観察項目で判定する。

| 観察項目 | 高 | 低 |
|---|---|---|
| 選択肢の数 | 3 つ以上の現実的代替案を比較 | 1 つの選択肢だけ語る |
| トレードオフ言語化 | 「A は速いが B は柔軟」のように軸を出す | 「A が良かったから」 |
| 定量と定性 | 数値根拠と組織根拠の両方 | 片方のみ |
| 制約への感度 | 当時の制約（時間・人・スキル）を認識 | 制約を語らず理想論 |

**スコアリング基準**:
- 3: 4 項目すべて高
- 2: 3 項目高
- 1: 2 項目以下
- 0: 選択肢を語らず「正解」のみを語る

---

## 3. 射程スコア（scope）

判断のスケールを段階値で記録する。**最も大きい射程の判断** を採用。

| 値 | 射程 | シグナル |
|---|---|---|
| 1 | タスク | 自分の手元の処理を最適化した |
| 2 | 機能 | 1 つの機能の設計判断を主導 |
| 3 | プロダクト | プロダクト全体に影響する選択を判断 |
| 4 | 事業 | 事業の収益・契約・コンプラに影響する判断 |
| 5 | 組織 | チーム編成・採用・文化・プロセスに影響する判断 |

パターンによって到達可能な最大射程は異なる（例: T-04 部分的な遅さの絞り込みは射程タスク〜機能）。パターン定義側に「期待射程レンジ」を持ち、それを超える射程が観察された場合はボーナス情報として `notes` に記録。

---

## 4. メタ認知スコア（meta_cognition）

文脈変化への感度を 4 つの観察項目で判定する（第 4 段の質問への応答で評価）。

| 観察項目 | 高 | 低 |
|---|---|---|
| 規模感度 | 「10 人規模ならやらない、100 人なら必要」と語れる | 規模に関わらず同じ判断と言う |
| フェーズ感度 | 「PMF 前 / 後で違う」「レガシー保守と新規で違う」 | フェーズの違いを感じない |
| バイアス自覚 | 「自分は〜寄りの判断をしがち」 | 自分の傾向を客観視できない |
| AI 影響感度 | 「2024 年以降なら AI を使って違う判断ができる」 | AI の登場を判断軸に入れない |

**スコアリング基準**:
- 3: 4 項目すべて高
- 2: 3 項目高
- 1: 2 項目以下
- 0: 第 4 段で詰まり、または「今でも同じ」と言い切り

---

## 5. AI 活用スコア（ai_literacy）

AI 活用横断軸の問いに対する応答で判定する。

| スコア | 兆候 |
|---|---|
| 3 | AI を「使う / 使わない」の境界を文脈で判断できる、失敗モードを認識している、自分のチームでの導入経験が具体的 |
| 2 | AI を使った経験はあるが境界が曖昧、AI で何が変わるかは語れる |
| 1 | AI を使ったことがない、または「使えば全部解決」「絶対使わない」の極端な立場 |
| 0 | 第 4 段の AI 観点問いで詰まり、応答なし |

A カテゴリ（A-01〜A-06）のパターンでは、ai_literacy が中核スコアとなる。それ以外のカテゴリでは横断軸として補助的に評価。

---

## 評価の二重スキーム

```
LLM 自動評価 (evaluateAnswer ツールが出力)
  ↓
assessment_answer.llm_evaluation JSONB
  { authenticity, judgment, scope, meta_cognition, ai_literacy, notes }

創業者手動評価 (admin UI で後から付与)
  ↓
assessment_answer.manual_evaluation JSONB
  { authenticity, judgment, scope, meta_cognition, ai_literacy, notes, reviewed_at, reviewer }
```

Stage 1 の検証指標:
- LLM 評価と手動評価の **次元別一致度**（5 次元それぞれで相関を計算）
- 既知実力評価（既存の人事評価・面接結果）との **手動評価の相関**
- 既知実力評価との **LLM 評価の相関**

LLM 評価が信頼できると判断できた次元は、Stage 2 で手動評価を省略する。

---

## 詰まり判定と次パターン移行

LLM は以下の条件で「詰まり」を検知し、深掘りを止めて次パターンへ移行する：

| 条件 | 判定 | アクション |
|---|---|---|
| 第 1 段で「経験なし」と明示的に回答 | not_experienced | level_reached=0 で記録、次パターンへ |
| 第 2 段で時系列・固有性が出ない（2 回深掘りしても抽象応答） | shallow | authenticity=0-1 で記録、次パターンへ |
| 第 3 段で選択肢が 1 つしか出ない（代替案を聞いても答えなし） | single_option | judgment=0-1 で記録、第 4 段は省略 |
| 第 4 段で「今でも同じ」と即答 | rigid | meta_cognition=0-1 で記録、AI 観点だけ問うて次へ |

`assessment_answer.level_reached` で到達段階を 1-4 で記録、または 0（経験なし）。

## LLM プロンプトでの評価指示

`evaluateAnswer` ツールを呼び出す際、LLM には以下を必ず指示する：

```
1. 5 次元のスコアを必ず整数で出力（射程は 1-5、他は 0-3）
2. 各スコアの根拠を notes に短く記述（経験者の典型シグナルがいくつ観察されたか）
3. 判断に迷う場合は低めに評価（false positive を避ける）
4. 詰まりがあれば level_reached を正確に記録
5. 矛盾検知した場合は notes に明記、authenticity を下げる
```

LLM 出力を DB に書く前に Zod schema で検証（`security.md` の Layer 6 参照）。スコアが範囲外、必須フィールド欠落の場合はツール再呼び出し。

---

## 集約とヒートマップ（Stage 2 以降）

Stage 1 では各パターンの 5 次元スコアを生データのまま保存。集約・可視化は Stage 2。

Stage 2 で導入する集約軸:
- カテゴリ別平均（D / T / P / S / O / A の 6 軸）
- 射程分布（1-5 のヒストグラム）
- AI リテラシー分布
- 経験パターン数（level_reached >= 2 の数）

ヒートマップ表示は職種で分けず、経験の濃淡で可視化する（product.md の方針に準拠）。
