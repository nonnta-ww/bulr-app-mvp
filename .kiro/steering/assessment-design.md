# Assessment Design Philosophy

bulr の中核は **面接アシスタント型の対話型経験問診**。本ドキュメントは、状況パターンの設計原則・4 段階深掘りの意図・面接官支援哲学を凝縮する。
具体的な 57 パターン定義は `docs/02-questionnaire-patterns.md`、深掘り質問テンプレは `docs/03-probe-logic.md` を参照。

## 問診の中核原理

### 知識テストではなく、判断経験を測る

| 旧路線（知識テスト型） | 新路線（状況パターン型） |
|---|---|
| EXPLAIN を使ったことがあるか？ | 本番でクエリ性能が突然劣化し、原因特定からインデックス設計の見直しまで判断した経験 |
| マイクロサービスの設計経験は？ | モノリスのうち 1 機能だけが他の数倍のリソースを要するようになり、分割するか同居で吸収するか判断した経験 |

**測りたいもの**: 「AI を道具として、複合領域にまたがるシステムを成立させる総合判断力」
**測らないもの**: 暗記知識、コーディング速度、特定ライブラリの使用経験

### 評価軸の三層構造

- **広さ**: システム種別 / トラフィック・データ規模 / チーム規模 / フェーズ / 職能横断
- **深さ**: 設計判断 / トラブル対応 / 負債との対峙 / 組織判断 / AI 活用判断
- **意思決定の射程**: タスク → 機能 → プロダクト → 事業 → 組織

射程が広い人ほど AI 時代に希少価値が高い、という仮説で評価設計。

## 面接アシスタント型の哲学

### AI は黒子、人間が決める

bulr は AI が人間を置き換えるツールではない。優れた人間の判断を支援するツール。

UI と挙動の至るところで、この哲学を表現する：
- 「自分で次を聞く」が常に選択肢として存在
- LLM の提案は3候補に絞り、面接官に選ばせる
- 採用推奨コメントを LLM に出させない（あくまで観察と提案、判断は人間）
- 最終評価には創業者の手動評価を組み合わせる

### 候補者は bulr 画面を見ない

主たるユーザーは面接官。候補者情報（name / applied_role / background_summary）は面接官が新規セッション作成時に入力する。候補者からの削除請求は企業側機能で対応（Stage 3 以降、bulr 側に削除フローなし）。

データオーナーは企業側（面接官）。bulr は AI 面接アシスタントとして、企業の業務プロセスを技術的に支援する立場。

### 音声録音 + Whisper 文字起こし

入力モダリティはテキストではなく音声。面接の自然な対話を保ったまま、録音 → Whisper → LLM 分析のサーバーオーケストレーションで処理する。

Stage 1 はチャンクストリーミング不採用、「次の質問へ」または「自分で次を聞く」押下時にまとめて Whisper API に送信。30 秒程度の沈黙は許容（面接の質を上げる側面もある）。

話者分離はプロンプトで吸収（Stage 2 で Deepgram 等の専用 API 導入を検討）。

## 状況パターンの設計条件

良い状況パターンは以下 4 条件を満たす：

1. **状況が具体的** — 「経験あり/なし」が明瞭に答えられる粒度
2. **判断点を含む** — 単一の正解ではなく、トレードオフを伴う選択がある
3. **判別性がある** — 本当に経験した者と聞きかじりの者で、深掘りで差が出る
4. **5 年以上の中堅以上が遭遇する蓋然性** — ジュニアでは出会いにくい状況

**Stage 1 のパターン総数**: 57 個（D:15 / T:12 / P:8 / S:8 / O:8 / A:6）

### カテゴリと採番ルール

| カテゴリ | 接頭辞 | 内容 |
|---|---|---|
| 設計判断 | D-XX | アーキ選定、データ設計、API、認証、整合性、責務分離 |
| トラブル対応 | T-XX | 障害一次対応、原因究明、組織連携、観測性、再発防止 |
| 性能・規模 | P-XX | ボトルネック、クエリ、キャッシュ、スケール、容量計画 |
| セキュリティ・データ | S-XX | 脆弱性対応、機密情報、権限、コンプラ、移行、DR |
| 組織・プロセス | O-XX | 合意形成、リプレイス、プロセス、規約、育成、採用 |
| AI 活用 | A-XX | LLM 組込、RAG、エージェント、コスト/品質、評価ループ |

採番は不変。パターンを廃止する際も番号は再利用しない（受験データの過去比較を保つため）。

## 4 段階深掘りの設計意図

| 段 | 何を測る | 質問の性格 | 通過の兆候 | 詰まりの兆候 |
|---|---|---|---|---|
| 第 1 段 | 経験の有無 | 状況に直面したことがあるか | 状況を具体的に再現できる（固有名詞、時期、規模） | 抽象的な肯定、一般論への退避 |
| 第 2 段 | 経験の真贋 | どんな症状で、どう特定したか | 一貫した時系列、捨てた仮説、関係者の役割 | 時系列の破綻、数値の欠落、当事者の不在 |
| 第 3 段 | 判断力 | どう解決し、なぜそれを選んだか | 複数選択肢の比較、トレードオフの言語化、後悔の自覚 | 「正解」を語る、選択肢が単一、後悔がない |
| 第 4 段 | メタ認知 | 規模・文脈が違えば同じ判断をするか | 文脈差異への感度、自分の判断のバイアス自覚 | 「今でも同じ」と言い切る、規模感への鈍感 |

### 重要な原則

- **特定パターンで詰まることそのものが情報** — ペナルティではなく、ヒートマップの濃淡を作る素材
- **詰まったら無理に引き出さず、別パターンへ移る** — 自然対話を保つ
- **経験なしと正直に答えた場合はペナルティなし** — 自己申告の真贋を高める

## 面接フロー（30〜40 分の使い方）

```
[ 面接前 ]
   - 面接官が新規セッション作成（候補者情報入力 + 同意取得済みチェック）
   - bulr が候補者経歴に基づき planned_pattern_codes（優先パターン）を生成

[ 状態A 録音中 ]
   - 質問テキスト表示（LLM 候補①/②/③ または manual の場合は空）
   - MediaRecorder で録音開始
   - 進捗インジケータ表示（パターン数 / 経過時間）
   - 操作: [次の質問へ] のみ

[ POST /api/interview/turns/next ]
   1. multipart/form-data で audio blob 受信
   2. uploadToBlob → Vercel Blob 保存
   3. transcribeAudio → 文字起こし
   4. (manual ターンなら) splitInterviewerCandidate → 質問+回答分離
   5. analyzeTurn → このターンのシグナル + 到達段階推定 + pattern_match_confidence
   6. interview_turn DB insert
   7. パターン完了判定 → aggregatePatternCoverage → pattern_coverage upsert
   8. proposeNextQuestions → question_proposal DB insert
   9. レスポンス: { turn, coverage?, proposal }

[ 状態B 候補選択 ]
   - 直前ターンの文字起こし（折り畳み表示）
   - 評価サマリー（このターンで観察できたこと）
   - 3 候補表示:
     - 候補①: 深掘りを続ける（現在パターンの未カバー段階）
     - 候補②: メタ認知や別視点（現在パターンの最終段階、規模感や代替案）
     - 候補③: 必ず「次のパターンに進む」を含める（時間管理支援）
   - 操作: [①] [②] [③] [自分で次を聞く]

[ ループ終了条件 ]
   - planned_pattern_codes を一通りカバー、または
   - 経過時間が 40 分到達、または
   - 面接官が「面接終了」ボタン押下

[ POST /api/interview/finalize ]
   - 残り pattern_coverage を集計
   - generateSessionReport → ヒートマップ + サマリー生成
   - status='completed'、completed_at 記録

[ 面接後レポート画面（面接官向け）]
   - ヒートマップ表示（CSS 横棒、Stage 1 簡易版）
   - サマリーテキスト表示（5 次元別所感 + カテゴリ別カバレッジ + フリー質問総評）
```

## 「自分で次を聞く」の挙動

3 候補のいずれも選ばない場合、面接官は [自分で次を聞く] を押す。

- 押下と同時に **録音が即開始** される
- 面接官は質問を話す。候補者が回答する。[次の質問へ] を押す
- bulr は録音から「面接官の質問」と「候補者の回答」を `splitInterviewerCandidate` で分離
- LLM が文脈から最も近いパターンを推定:
  - **exact**: LLM 提案候補を選んだ場合（このルートでは出ない）
  - **inferred_high**: 文脈的に明確に特定パターンに該当
  - **inferred_low**: 関連パターンが推定できるが confidence 低
  - **off_pattern**: どのパターンにもマッピングできない（フリー質問）

これにより、面接官は **3 候補を採用するか、自分の判断で進めるか**を自由に選べる。LLM の提案はあくまでオプションであり、強制されない。

---

## フリー質問（規定外）の扱い

面接官の経験豊富さから、57 パターンに収まらない貴重な質問が出ることがある。これを許容する設計：

- `pattern_id = null`、`pattern_match_confidence = 'off_pattern'` で記録
- `off_pattern_summary` にフリー質問の要約を保存
- **評価集約（pattern_coverage）には含めない** — 5 次元スコアリングの対象外
- **session_report.summary_text に総評として反映** — 「規定外質問が N 件あった、内容: ...」

新パターンへの昇格判断は Stage 1 では別途検討（DB 直接閲覧で対応）。Stage 2 以降で admin の専用 UI 検討。

---

## AI 活用横断軸（全パターン共通）

bulr の独自性の核。各パターンの第 4 段最後に **AI 活用観点の問い** を差し込み、AI 時代の希少価値を引き出す。

差し込みパターン:
- 「このプロセスで AI を使えるとしたら、どこを任せて、どこを自分でやりますか？」
- 「AI ツールが使える今、同じ問題に対するアプローチは変わりますか？」
- 「逆に『絶対 AI には任せない』部分はどこですか、なぜ？」

セッション末（クロージング 5 分）の総括問い:
- 「今回話したいくつかの状況のうち、AI を使えば違う判断・解決ができたものはありますか」
- 「あなたが AI に任せたくない・任せられない判断は何ですか」
- 「これから AI 前提で開発するチームを作るなら、何を一番先に変えますか」

これらは LLM の `proposeNextQuestions` が候補生成時に自然と差し込む。

## LLM の振る舞い指針

### 自然対話を保つ工夫

- **オープンクエスチョン優先**: 「はい / いいえ」だけで終わらせない
- **続きを促す**: 「もう少し詳しく」「他には？」を多用
- **相槌と要約**: 「なるほど、〜という状況だったということですね。それで…」
- **時間管理**: 1 パターンあたり 5〜7 分で次へ
- **詰まり時の救済**: 「経験がなくても問題ありません、別の状況に移りましょう」と明示的に逃げ道を作る
- **採用推奨コメントを出さない** — あくまで観察と提案、判断は面接官

### 矛盾検知のヒューリスティクス

詰問せず、別の角度から確認質問を投げる：
- **時系列の破綻**: 「もう少し具体的な数値で言うと？」
- **規模の不一致**: 「他のメンバーはどう関わっていましたか？」
- **当事者の不在**: 「決定権を持っていたのは誰ですか？」
- **後悔の欠落**: 「今ならどう変えますか？」

### 詰まり判定 4 種

LLM は以下の条件で「詰まり」を検知し、深掘りを止めて次パターンへ移行する：

| 条件 | 判定 |
|---|---|
| 第 1 段で「経験なし」と明示的に回答 | not_experienced |
| 第 2 段で時系列・固有性が出ない（2 回深掘りしても抽象応答） | shallow |
| 第 3 段で選択肢が 1 つしか出ない（代替案を聞いても答えなし） | single_option |
| 第 4 段で「今でも同じ」と即答 | rigid |

これらは `pattern_coverage.stuck_type` に enum で記録。

## パターン磨き込みの運用方針

問診設計は **常に未完成**。受験データから継続的に改善する：

- **追加**: 受験者から「こういう状況の問いがなかった」と指摘された判別性の高い状況
- **統合**: 高相関で判別性が重複しているパターン（A の経験者は B も経験している率が極端に高い等）
- **改廃**: 受験率・判別性・採用後パフォーマンスとの相関で評価し、廃止判断
- **AI カテゴリ（A-XX）**: 技術トレンドの変化が速いため、3 ヶ月ごとに見直し（最優先で更新する領域）
- **フリー質問からの昇格**: 規定外質問が複数セッションで類似パターンを示した場合、新規 D-16 / T-13 等として追加検討

パターン番号は不変。廃止しても再利用せず、`assessment_pattern.is_active = false` で論理的に休眠させる。

## Stage 1 で扱わないパターン拡張

- 職種拡張（フロントエンド / SRE / PdM 領域のパターン）
- 業界文脈（決済、医療、組み込み、金融）
- 規模文脈（0→1 / 大規模 / レガシー再生でのフェーズ別出題分岐）
- 多言語対応（英語、ベトナム語）

これらは Stage 2 以降で順次追加。Stage 1 はバックエンド × 日本語のみで検証に集中。

## ベトナム人候補者の面接（Stage 1 の現実的扱い）

ベトナムオフショア会社の協力で 20-30 人の面接データを取得予定。Stage 1 では：

- 面接の言語：面接官が現地語（日本語可能なベトナム人エンジニアのみ）または英語で実施
- 候補者の応答もそのまま録音 → Whisper は多言語対応のため自動的に対応言語で文字起こし
- LLM のシステムプロンプトは日本語ベース、応答も日本語
- 面接官が日本語以外で質問した場合、`split_interviewer_candidate` で分離後、LLM が翻訳を踏まえて分析
- ヒートマップ・サマリーは日本語で生成（創業者の検証用）

文化的ギャップによる判断力評価のフェアネスは創業者が手動評価でカバー。
