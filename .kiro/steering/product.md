# Product Overview

## ビジョン

**bulr** — _面接官の隣に座る、AI 面接アシスタント_

> エンジニア採用面接で、面接官の発言を最小限に抑えながら、候補者の実務判断力を引き出す質問を提案し、回答を構造化して評価する。

書類選考に時間を奪われる採用責任者と、書類では実力が伝わらないエンジニアの両方を救う。

## 主たる利用者

**採用面接の面接官** — 事業会社の VPoE / EM / 採用責任者、スタートアップ CTO・創業者、採用担当のシニアエンジニア。**候補者は bulr を直接使わない**。bulr は面接官の側に存在し、判断を支援する黒子。

将来的（Stage 3 以降）に候補者直接対話型の追加検討余地はあるが、Stage 1 では面接アシスタント型のみ。

## 評価対象

- 中堅以上のエンジニア（実務経験5年以上目安）
- 設計判断・トラブル対応・組織判断を経験している層
- アウトプット（GitHub・登壇・ブログ）が少なく、既存サービスで可視化されない層

**構造的に外す**:

- ジュニア層・新卒（AI 時代の構造変化により別市場）
- 既にアウトプットで可視化されている上位数 % の有名エンジニア

## 主要価値

| 価値         | 内容                                                                       |
| ------------ | -------------------------------------------------------------------------- |
| 面接官支援   | 経験豊富な面接官の判断を支援、普通の面接官にも使える道具                   |
| 経験の可視化 | 書類に書ききれない実務判断の経験を、4 段階の深掘り対話で引き出す           |
| 真贋の判別   | 経験者・聞きかじり・盛りを、深掘りでの一貫性・固有性・後悔の有無で見分ける |
| 多次元評価   | 職種で分けず、広さ × 深さ × 意思決定射程で経験の濃淡をマッピング           |

## 競合ポジショニング

```
Findy / LAPRAS = 「何を作ってきたか」の可視化（アウトプットベース）
paiza / AtCoder = 「何が書けるか」の可視化（コーディング能力）
SPI 系          = 「地頭」の可視化（一般能力）
bulr            = 「何を判断してきたか」の可視化（実務経験ベース）+ 面接官支援
```

競争優位の 3 つの源泉:

1. **対話型経験問診** — 4 段階の深掘りで経験の真贋を判別、書くのが苦手な実力者も可視化
2. **構造化された状況パターン** — 設計判断・トラブル対応・性能・セキュリティ・組織・AI 活用の 6 カテゴリ × 57 パターン
3. **5 次元スコアリング** — 真贋・判断力・射程・メタ認知・AI 活用リテラシーで多面的に評価

## なぜ面接アシスタント型か（v1 候補者対話型からの転換理由）

1. **対話品質のリスクを切り離せる** — 人間の面接官が対話の主役、LLM が多少不器用な質問でも面接官が咀嚼して投げ直せる
2. **検証データの質が高い** — 候補者は人間相手に本気で答える、AI 相手のテキスト入力より圧倒的にクリーン
3. **市場が受け入れやすい** — 「人間が面接することが重要」観念に整合、導入心理的ハードルが低い
4. **採用責任者の本当の痛みに直接刺さる** — 「面接で何を聞けば実力が分かるか分からない」を直接解決
5. **創業者の優位性が最大化される** — 26 年の経験 × 年間 800 書類が結晶化した「問診パターン」を、優れた面接官の道具として提供

## Stage 1 主要機能（MVP プロトタイプ）

```
[面接官向け]
- Magic Link サインイン (Better Auth)
- セッション一覧（自分が担当した面接）
- 新規セッション作成（候補者情報入力 + 同意取得済みチェック）
- 面接中 UI:
  - 状態A（録音中）：質問表示 + 進捗インジケータ + [次の質問へ]
  - 状態B（候補選択）：直前ターンの文字起こし + 評価サマリー + 3 候補 + [自分で次を聞く]
- 面接後レポート：ヒートマップ + サマリーテキスト（5 次元別所感 + カテゴリ別カバレッジ + フリー質問総評）

[創業者向け（管理画面、apps/web 同居・ADMIN_ALLOWED_EMAILS 許可メール検査）]
- 全セッション一覧
- 回答全文確認（chat_message 時系列 + assessment_answer 4 段階回答）
- 5 次元手動評価入力フォーム
- LLM 評価との突合表示
- CSV / JSON エクスポート

[問診の構造]
- 6 カテゴリ × 57 状況パターン (D/T/P/S/O/A)
- 4 段階深掘り (経験有無 → 真贋確認 → 判断力 → メタ認知)
- AI 活用観点の横断軸（全パターン共通）
- フリー質問（規定外質問）の許容と総評反映
```

**Stage 1 スコープ外**: 候補者向け UI、リアルタイム文字起こし、話者分離 API、企業向けダッシュボード、ヒートマップ可視化の本格化（Stage 2）、複数職種、ドラクエ的パーティ編成、課金、求人・応募管理、人材紹介、教育コンテンツ

## Stage 1 検証ゴール（3 ヶ月）

> **「バックエンドエンジニア向けの面接アシスタントを作り、創業者および協力面接官 5-10 人が、ベトナム人 20-30 人 + 日本人 10-20 人の面接で実際に使い、問診パターンに基づく面接結果と、面接官の独自判断との一致度を確認する」**

成功指標:

- 面接官完走率: 予定時間内（30-40 分）で 12-15 パターンをカバー
- bulr の提案有用度: 3 候補から 1 つ以上を選ぶ率 70% 以上
- 評価の一致度: bulr ヒートマップと面接官独自判断の相関が高い
- 継続意向: ベータ参加面接官の 70% 以上が「使い続けたい」

3 ヶ月後の意思決定:

- 問診は機能しているか / 企業は価値を感じているか / 自分はやり続けたいか
- 3 つ揃えば本格投資、1 つでも欠ければ方向転換

## 中長期ロードマップ

### Stage 2 (Month 4-12): 本格立ち上げ

- フロントエンド・SRE/インフラ・PdM 寄り領域に拡張
- リアルタイム文字起こし（チャンク単位ストリーミング）導入
- 先読みでの質問生成（状態A 中に状態B 候補先読み）
- 話者分離 API（Deepgram など）導入
- ヒートマップの本格的な可視化
- 候補者向けレポートの生成
- 企業向け月額（10-30 万）でマネタイズ開始
- 最初の有料顧客 5-20 社

### Stage 3 (Year 2): 両面化とインフラ化

- 候補者直接対話型の追加（v1 で保留した方式）
- マルチテナント機能（ワークスペース）
- 企業向けダッシュボード
- 求人管理、応募管理、業務委託マッチング
- 「bulr スコア」の業界用語化
- API 公開
- **人材紹介事業の参入判断**

### Stage 4-5 (Year 3-5): パーティ編成 & 標準化

- 業界標準化、アトラクト・リテンション支援、組織編成コンサル
- 人材流動性インフラへ

## 収益モデル方針

```
取らない: 教育コンテンツ作成（労力対効果悪い）
取る:    企業向け SaaS 月額、API 利用料、組織診断コンサル、データ販売
保留:    人材紹介（Stage 3 以降のオプション、データ充実後に「データに基づく紹介」として）
```

Stage 1 は検証専念、マネタイズなし。

## データオーナーシップ（重要）

bulr のデータオーナーは **企業側**（面接官 = `interviewer_id`）。bulr は AI 面接アシスタントとして、企業の業務プロセスを技術的に支援する立場。

候補者からの削除請求は **企業側機能** で対応（Stage 3 以降）。Stage 1 では bulr 側に削除フローを設けない。ただし音声ファイルのみは 30 日後自動削除（プライバシー配慮 + ストレージ負担軽減）。

## メンタルモデル：ドラクエのパーティ比喩（重要）

採用観の核：

1. 全部できる人は存在しないのに、求人票はそれを求める
2. スキル分布の解像度が低く、戦士に魔法を求めるミスマッチが起きる
3. クエスト（事業フェーズ）に応じてパーティを組み替えるべきなのに、組織は固定的

bulr は将来的に「ドラクエ的な組織編成インフラ」を目指す。Stage 1 では候補者個人の可視化に集中。

## プロダクト哲学

- **AI は黒子、人間が決める**：「自分で次を聞く」が常に選択肢、最終評価には創業者の手動評価を組み合わせる
- **書類では伝わらない実力を救う**：情報発信が苦手なエンジニアが、面接で実力を発揮できる仕組み
- **ドラクエのパーティ思考**：データ設計はパーティ編成の方向性を意識（Stage 1 では UI に現れない）
- **AI 時代の実務判断力に焦点**：コードを書く能力ではなく「何を作るか、どう判断するか」

## 用語集

| 用語                  | 定義                                                                                                                          |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 面接官（interviewer） | bulr の主たるユーザー。Magic Link サインインして使用                                                                          |
| 候補者（candidate）   | 評価対象。bulr 画面は見ない、面接官との対面/オンライン面接に参加                                                              |
| 状況パターン          | 実務判断を引き出すための具体的状況、6 カテゴリ × 57 個                                                                        |
| カテゴリ              | D=設計判断 / T=トラブル対応 / P=性能・規模 / S=セキュリティ・データ / O=組織・プロセス / A=AI 活用                            |
| 4 段階深掘り          | 経験有無 → 真贋確認 → 判断力 → メタ認知 の段階的な質問構造                                                                    |
| ターン                | 1 質問 + 1 回答（または manual ターン）。録音 → Whisper → LLM 分析                                                            |
| パターンカバレッジ    | 1 セッション × 1 パターンの集約評価（複数ターンを統合）                                                                       |
| フリー質問            | 57 パターンに該当しない、面接官が独自に思いついた質問。`pattern_id=null` で記録、評価集約には含めず session_report に総評反映 |
| 真贋スコア            | 経験が本物か（具体性・時系列・関係者・捨てた仮説・後悔の有無） 0-3                                                            |
| 判断力スコア          | トレードオフを認識しているか（選択肢の数・軸の言語化・定量定性） 0-3                                                          |
| 射程スコア            | 判断のスケール（タスク / 機能 / プロダクト / 事業 / 組織） 1-5                                                                |
| メタ認知スコア        | 文脈変化への感度（規模・フェーズ・バイアス自覚） 0-3                                                                          |
| AI 活用リテラシー     | AI を使う / 使わない境界の判断軸 0-3                                                                                          |
| AI 活用横断軸         | 全パターンで第 4 段最後に差し込む「AI に任せる / 任せない判断」の問い                                                         |
| 状態A                 | 録音中、質問表示 + [次の質問へ]ボタンのみ                                                                                     |
| 状態B                 | 候補選択中、3 候補 + [自分で次を聞く]ボタン                                                                                   |
